Native execution engines · STAR analysis

Velox

Every analytics system — Presto, Spark, every warehouse and stream processor — rebuilds the same execution layer: the columnar memory format, the vectorized expression evaluator, the hash join, the spilling aggregation. Velox is Meta's bet that you should write that layer once, as a reusable C++ library, and let engines plug it in. It is not a database: no SQL parser, no optimizer, no dataframe API. It takes a fully optimized plan and runs it, fast. That single decision is why Velox now sits underneath Presto (as Prestissimo) and Apache Spark (via Gluten), developed by Meta, IBM, Intel, Microsoft, ByteDance and others. Here is what's actually inside, and where a "just a library" promise gets complicated.

In short

One idea — factor the execution engine out of the database and ship it as a library — and an unusually broad coalition of companies building on top of it.

What it is

An open-source C++ library of reusable, high-performance data-processing components: an Arrow-compatible Vector layer, a vectorized expression evaluator, relational Operators (scans, joins, aggregations, sorts), connectors for ORC/Parquet/Nimble and S3/HDFS/GCS, and resource management (memory arenas, spilling, caching). You feed it an already-planned query; it executes. Created by Meta, Apache-2.0.

Why it's interesting

It dissolves the boundary between "engines." The same operator code that runs a Presto query now runs a Spark SQL stage under Gluten — so a SIMD speedup or a join-spilling fix lands everywhere at once. Consistent semantics, consistent performance, one place to optimize. That consolidation is the real product, more than any single benchmark number.

Biggest concerns

It is a developer's library, not a user's tool: no SQL parser, optimizer, or dataframe layer, so you can't "run Velox" — you integrate it. It demands modern SIMD (bmi, bmi2, f16c — recent x86 instruction-set extensions), a C++ toolchain, and real systems expertise. There are no tagged releases — you build from a fast-moving main — and the surface area (1,500+ open issues) is large.

Verdict

One of the most consequential pieces of open-source data infrastructure of the decade — not because of a headline benchmark, but because it makes "the execution engine" a shared, reusable component instead of something every system reinvents. If you are integrating or accelerating an analytics engine in C++, Velox is the substrate to build on, and the partner list (Meta, IBM, Intel, Microsoft, ByteDance, Voltron Data) means it isn't a single-vendor risk. If you are an end user who just wants to run SQL, Velox is invisible plumbing — you'll meet it through Prestissimo or Gluten, never directly. Treat "build from main, no releases" and the SIMD requirement as the real costs of entry.

1 · Situation — every engine rewrites the same layer

By the early 2020s, the analytics landscape had a quiet duplication problem.

Look inside Presto, Spark, ClickHouse, a dozen cloud warehouses, and a handful of stream processors, and you find the same machinery rebuilt over and over: a columnar in-memory format, a vectorized expression evaluator, a hash-join implementation, a group-by that spills to disk under memory pressure, a set of scalar and aggregate functions, readers for Parquet and ORC, adapters for S3 and HDFS. Each system wrote its own version, in its own language, with its own bugs, its own performance cliffs, and its own subtly different semantics for what NULL + 1 or an overflowing SUM should do.

That duplication is expensive twice over. It is expensive to build — every team re-implements vectorized joins from scratch. And it is expensive to improve — a clever SIMD decoding trick or a smarter spilling strategy has to be re-discovered and re-applied in every engine independently. The state of the art advances in one codebase and the others wait years to catch up.

Velox is Meta's answer: stop rewriting the engine. Meta ran several internal systems — Presto, Spark, an internal stream processor, ML feature-engineering pipelines — each with its own execution layer. Factoring that layer into a single, shared C++ library meant one place to fix bugs, one place to optimize, and — crucially — consistent behavior across products. Open-sourcing it in 2021 turned that internal consolidation into an industry one.

The old way — every engine its own

Presto had its engine, Spark had Tungsten, ClickHouse had its own. Each re-implemented vectors, expressions, joins, spilling — with divergent semantics and divergent speed.

The duplication tax

A SIMD or spilling improvement helps one engine and stops there. The frontier moves in one codebase while the rest wait years to copy it.

Velox — write it once

One C++ library of execution components, reused across engines. Fix or optimize once; every consumer benefits. Consistent semantics by construction.

The wager

The risky bet: that execution is generic enough to share. The payoff is the partner list — Meta, IBM, Intel, Microsoft, ByteDance — all building on the same core.

2 · Task — a library, deliberately not a database

Velox's scope is defined as much by what it refuses to do as by what it does.

The README is unusually disciplined about the boundary: Velox "does not provide a SQL parser, a dataframe layer, or a query optimizer," and "is usually not meant to be used directly by end-users." It takes a fully optimized query plan as input and performs the described computation. Everything above the plan — parsing, binding, cost-based optimization, the user-facing API — stays the responsibility of the engine that embeds Velox.

That leaves a precise contract. The host engine (Presto, Spark, your own) parses and optimizes the query, then hands Velox a PlanFragment: a tree of operators with resolved types and expressions. Velox builds the operator pipeline, pulls data through it in vectorized batches, manages memory and spilling, and streams results back. The promise is fourfold:

  1. Reusable — the same components serve batch, interactive, streaming, and AI/ML feature workloads.
  2. Extensible — engines plug in their own types, functions, operators, file formats, storage adapters, and wire serializers.
  3. High-performance — fully vectorized, SIMD-accelerated, with a memory model built for tight control and spilling.
  4. Consistent — function semantics follow Presto and Spark, so an engine can match its reference exactly.

The thesis of this report: promises (1)–(3) are delivered and genuinely impressive; promise (4) is the subtle, load-bearing one — Velox ships two function namespaces (Presto and Spark semantics) precisely because "consistent" means "consistent with whichever engine you're impersonating," and getting that exactly right is most of the integration work.

3 · Action — what's actually in the library

Seven lenses: the component map, vectors, the expression engine, the operator/driver model, the many engines on top, extensibility, and project health.

3.1 The shape — eight components, one shared object

Velox is organized as a set of cooperating components, each individually reusable:

  • Type — a generic type system covering scalars and nested/complex types (structs, maps, arrays).
  • Vector — the Arrow-compatible columnar memory module (detailed below).
  • Expression Eval — a fully vectorized evaluator that runs expression trees over vectors.
  • Functions — vectorized scalar, aggregate, and window functions, in Presto-semantic and Spark-semantic variants.
  • Operators — scans, projections, filters, group-by, ordering, shuffle/exchange, and hash / merge / nested-loop joins, unnest, and more.
  • I/O connectors — file formats (ORC/DWRF, Parquet, Meta's Nimble) and storage adapters (S3, HDFS, GCS, ABFS, local).
  • Network Serializers — pluggable wire protocols, including PrestoPage and Spark's UnsafeRow.
  • Resource Management — memory arenas, buffer management, tasks/drivers, CPU and IO thread pools, spilling, and caching.

The point of the decomposition is that you don't have to take all of it. A team can reuse just the Vector layer for an Arrow-native data path, or just the expression evaluator, or the whole operator set. That à-la-carte property is what makes Velox a library rather than a framework.

3.2 Vectors — Arrow-compatible, and encoding-aware

The Vector module is the foundation everything else stands on. It is an Arrow-compatible columnar layout, which means zero-copy interop with the broader Arrow ecosystem — but Velox adds encodings and a lazy-materialization pattern that go beyond a plain flat buffer. The four headline encodings:

  • Flat — the plain case: a contiguous buffer of values plus a null bitmap.
  • Dictionary — an array of indices into a smaller values vector. A low-cardinality column (say, a country code repeated across a billion rows) stores each distinct string once and a cheap integer per row.
  • Constant — a single value standing in for an entire column, for literals and broadcast cases — one value, not N copies.
  • Sequence / RLE — run-length encoding for long stretches of repeated values.

Encodings aren't just a storage trick; they flow through execution. A filter or expression can often evaluate against a dictionary's compact value set and produce another encoded vector, so the savings compound across operators instead of being flattened away at the first step. Combined with lazy materialization — don't decode a column until an operator actually needs it — this is a large part of where Velox's speed comes from.

3.3 The expression engine — vectorized, with constant folding and short-circuiting

Velox evaluates whole expression trees over vectors, not row-by-row. An expression like upper(trim(name)) = 'ADA' AND age > 30 compiles to a tree the evaluator walks once per batch, applying each function across the vector with SIMD where the function supports it. The engine does the optimizations you'd expect from a real compiler-ish layer: constant folding, common-subexpression elimination, null short-circuiting (skip work on rows already known to be null), and peeling encodings so a function runs on a dictionary's values rather than every row. Functions come in two flavors — a "simple function" API (write scalar logic, get vectorization for free) and a "vectorized function" API (hand-tune the batch loop) — which is how Velox keeps the common case easy and the hot path fast.

3.4 The driver/operator model — pull-based vectorized pipelines

Execution is organized as a Task split into pipelines, each run by one or more Driver threads. A driver threads a chain of operators together and pulls data through them a batch at a time: each operator takes a RowVector (a columnar batch) from upstream, does its work, and hands a RowVector downstream. Operators that need to block — a hash join building its table, an aggregation accumulating, an exchange waiting on the network — yield cleanly so the thread can do other work.

This is the layer that used to live inside each engine and now lives once, here. The resource manager underneath it — arenas, buffer pools, the spilling and caching primitives — is what makes the operators production-grade rather than textbook: a join that exceeds its memory budget spills instead of crashing, and a hot scan can be served from Velox's async data cache instead of re-reading remote storage.

3.5 One library, many engines

The reason Velox matters in practice is the set of systems that now run on it. The same operator and expression code is shared across very different frontends:

Prestissimo replaces Presto's Java workers with C++ workers built on Velox. Gluten offloads Spark SQL stages to Velox (the subject of the companion post). Meta and partners also use it for stream processing and ML feature engineering. The strategic consequence is what the README's coalition implies: an optimization contributed by Intel for a Presto workload can speed up a Spark job at ByteDance, because it lands in the shared core.

3.6 Extensibility — eight ways to specialize

Velox is explicit that integrators will need to specialize it, and it names eight extension points: custom types, simple and vectorized functions, aggregate functions, window functions, operators, file formats, storage adapters, and network serializers. This is how the same core serves engines with genuinely different semantics: Spark's UnsafeRow wire format and Presto's PrestoPage are both just registered serializers; Spark-semantic and Presto-semantic functions are both just registered function sets. The extension surface is the product — without it, "reusable across engines" would be a slogan.

3.7 Project health — broad backing, fast main, sharp edges

Velox is unusually well-supported for an infrastructure library: created by Meta and developed in partnership with IBM/Ahana, Intel, Voltron Data, Microsoft, ByteDance, and others — a coalition that de-risks the bus-factor question most open-source engines carry. The flip side is engineering reality: there are no tagged releases, so you build from a fast-moving main; the CPU must support modern SIMD (bmi, bmi2, f16c, with avx/avx2/sse on Intel and Neon on ARM used where available); and the issue tracker carries 1,500+ open items, which is the texture of a large, actively pushed C++ codebase rather than a sign of neglect. The active blog (SIMD Parquet decoding, FlatMapVector for AI/ML preprocessing) shows where the frontier is moving: lower-level kernel performance and AI data pipelines.

The win that matters most in production is the one that doesn't show up in a single benchmark: you stop maintaining your own execution engine. A team integrating Velox trades the cost of writing and tuning vectorized joins, spilling aggregations, and Parquet readers for the (real, non-trivial) cost of a C++ integration against a moving target. For an organization running analytics at scale, that's usually a good trade — which is precisely why the consortium exists.

The caveats are equally concrete. Velox is a heavy dependency: a C++ build, SIMD requirements, and an API that tracks main rather than a stable release line. It assumes you already have a planner — if you don't, Velox gives you nothing user-facing on its own. And matching your engine's exact semantics (especially around nulls, overflow, decimals, and timestamps) is detailed, unglamorous work that the library enables but does not eliminate.

5 · Takeaways — how to read Velox in 2026

Five calibrated conclusions for an engineer evaluating it.

The real product is consolidation

Velox's value isn't a benchmark; it's that execution becomes a shared component. Judge it by the coalition building on it and by "one fix helps everyone," not by any single speedup figure.

It's a library for engine builders

If you're integrating or accelerating a compute engine in C++, this is the substrate. If you want to run SQL, you want Prestissimo or Gluten — you'll get Velox underneath without touching it.

Encodings and laziness are where the speed lives

Dictionary/constant/RLE vectors plus lazy materialization let work shrink as it flows through the pipeline. Understanding that model is the key to understanding Velox's performance.

Budget for "build from main" and SIMD

No tagged releases means you pin a commit and track a moving API; the SIMD floor (bmi2, f16c) means older or constrained CPUs are out. These are the genuine costs of entry.

Semantics are the integration

The two function namespaces exist because "fast" is easy and "bit-for-bit like Spark/Presto" is hard. Plan most of your integration effort around matching reference semantics, not around performance.
