In short

Five bullets the rest of this report defends with code, citations, and benchmarks.

What it is

A local-first TypeScript MCP server + CLI that turns any codebase into a SQLite knowledge graph — symbols, calls, imports, framework routes — then exposes 10 typed tools so an AI coding agent answers structural questions in a handful of calls instead of dozens of grep/Read round-trips. MIT.

Why it's interesting

Ships a bundled Node runtime so there are zero native addons (no better-sqlite3 rebuild dance), uses Node 22.5's built-in node:sqlite with WAL + FTS5, and synthesizes edges across boundaries that static tree-sitter parsing can't cross — React setState→render, JSX child→component, Swift↔Objective-C bridging, React Native legacy + TurboModules + Fabric, Expo modules, Spring + MyBatis XML.

Biggest concerns

One person wrote 92% of the code (317 of 377 commits). Per-process daemon model gives up on Windows network shares and some WSL2 layouts where WAL fails to enable. Agent-side gains depend on whether the host model picks the right tool — the maintainer's own notes admit codegraph_trace adoption is fragile across models.

1 · Situation — agents that can read but can't navigate

A pre-built index of symbols and call sites is what compiler IDEs have solved for thirty years. Coding agents are only now catching up.

Through 2025 the AI coding assistant market converged on a workflow that looks the same in Claude Code, Cursor, Codex CLI, Aider, opencode, and Hermes: a host LLM is given a small palette of file tools — Read, Grep, Glob, Bash — and an MCP plug-point for everything else. Ask any of them "how does the extension host talk to the main process in VS Code?" and the agent fans out: list files, grep for likely names, read 20 files, often spawn an "Explore" sub-agent that reads 30 more, and finally answer. The cost on a large repo runs into dollars per query and minutes per answer.

The shape of the problem is exactly what compiler IDEs have solved for thirty years — a pre-built index of symbols, definitions, call sites, and inheritance — except the index has to be (a) cheap enough to maintain incrementally on a developer's laptop, (b) reachable through a wire format the agent already speaks, and (c) confident enough that the agent stops looking after the first answer. Several projects have shipped against that gap in the last twelve months:

Serena — semantic via LSP

oraios/serena (24.7K ⭐, Python, MIT) wraps language-server-protocol adapters to give MCP-speaking agents semantic retrieval and editing — relies on a live LSP per language.

CodeGraphContext — graph DB

CodeGraphContext/CodeGraphContext (3.5K ⭐, Python) indexes a repository into a separate graph database and surfaces the result through MCP — heavier install, richer query surface.

codebase-memory-mcp — static binary

DeusData/codebase-memory-mcp (2.8K ⭐, C) ships as a single static binary, claims 155 languages and "99% fewer tokens" — different language reach, less detailed claim auditing.

code-review-graph & Aider repo-map

tirth8205/code-review-graph (17.6K ⭐) frames the same local-first graph idea around code reviews. Aider-AI/aider (45.5K ⭐) bundles a tree-sitter "repo-map" inside a chat agent rather than shipping as a standalone graph server.

colbymchenry/codegraph (31k ⭐ in four months, MIT, TypeScript) stakes a specific position in that bake-off: stay in TypeScript so the same code is the library, the CLI, and the MCP server; ship a vendored Node runtime so installation is one curl; speak only MCP + stdio so it works the same in every supported agent. That positioning is what this report unpacks.

2 · Task — what CodeGraph commits to solving

Four concrete deliverables, stripped of marketing.

Stripped of the README's pitch, the project commits to four concrete deliverables:

  1. A deterministic knowledge graph from tree-sitter, not LLM summaries. Every node and edge is derived from the AST so the index is reproducible and stable across runs.
  2. A daemon that an agent can talk to without changing how it codes. Drop in the MCP config, restart the agent, the tools just appear.
  3. A tool surface tuned for "I'm done — answer" rather than "let me explore more". The tools inline call-site source, hop bodies, and registration sites so a single trace call ends the investigation.
  4. Coverage of the dynamic-dispatch boundaries grep cannot cross. Callbacks, setState→render, JSX child→component, Swift @objc bridging, React Native bridges, Spring @Value + MyBatis XML — the places where flow questions actually break.

The actual gap it fills: a vendor-neutral, locally-hosted, zero-config code-intelligence MCP server that the model can use the moment it's installed. No vector DB to provision, no embedding cost, no telemetry. The price for "local-first + zero-config" is that everything is static: no LLM enrichment of nodes, no learned ranking — what you see is what tree-sitter saw, augmented by some carefully-scoped synthesizers.

3 · Action — how it actually works

Reading the code, not the README. Nine lenses: architecture, storage, extraction, resolution, MCP tool surface, auto-sync, shared daemon, multi-agent installer, tests.

3.1 Architecture at a glance

The package is one TypeScript codebase that serves three roles: a Node library (import CodeGraph from '@colbymchenry/codegraph'), a CLI (codegraph init|index|sync|status|query|files|context|callers|callees|impact|affected|serve --mcp), and the MCP server itself (src/index.ts L1-80 re-exports the whole surface so the bin and the MCP entrypoint share a single class).

The pipeline the maintainer documents in CLAUDE.md L36-46 is the right mental model:

3.2 Storage — Node's built-in SQLite, WAL, FTS5

The schema in src/db/schema.sql L20-67 is the smallest possible thing that can hold a knowledge graph:

CREATE TABLE nodes (
    id TEXT PRIMARY KEY, kind TEXT, name TEXT, qualified_name TEXT,
    file_path TEXT, language TEXT, start_line INTEGER, end_line INTEGER,
    docstring TEXT, signature TEXT, decorators TEXT, …
);
CREATE TABLE edges (
    id INTEGER PRIMARY KEY, source TEXT, target TEXT,
    kind TEXT, metadata TEXT, line INTEGER, provenance TEXT,
    FOREIGN KEY (source) REFERENCES nodes(id) ON DELETE CASCADE
);
CREATE TABLE files (
    path TEXT PRIMARY KEY, content_hash TEXT, language TEXT,
    size INTEGER, modified_at INTEGER, indexed_at INTEGER, …
);
CREATE VIRTUAL TABLE nodes_fts USING fts5(
    id, name, qualified_name, docstring, signature,
    content='nodes', content_rowid='rowid'
);

Two design choices are worth flagging.

Edges carry provenance and metadata as first-class columns (L43-55). Every synthesized edge — anything that wasn't directly extracted from the AST — is tagged provenance:'heuristic' with a metadata.synthesizedBy string. The MCP trace tool renders these inline as "↳ registered at App.tsx:3148" so the agent (and the human reading its output) can see how a hop got into the graph. This is the difference between a synthesizer that helps and one that lies; CodeGraph commits a budget to the "explain yourself" half. Coverage: callback, event-emitter, react-render, flutter-build, cpp-override, interface-impl, jsx-render, vue-handler, rn-event-channel, fabric-native-impl, mybatis-java-xml — all in src/resolution/callback-synthesizer.ts.

Indexes are deliberately narrow. Line 126-130 carries the comment "idx_edges_source / idx_edges_target are intentionally omitted — the (source, kind) and (target, kind) composites below cover the corresponding source-only / target-only lookups via SQLite's left-prefix scan, so the narrow indexes are dead weight on writes. Migration v4 drops them on existing databases." Someone has actually measured the write penalty.

The storage backend, in src/db/sqlite-adapter.ts L43-118, is a 75-line wrapper over Node 22.5's built-in node:sqlite. The decision is documented in BUNDLING.md L3-12:

Because Node 22.5+ has a built-in real SQLite (node:sqlite, with WAL + FTS5), bundling Node means: no native build — better-sqlite3 is gone, so there are zero native addons to compile or rebuild. No wasm fallback — and therefore no more database is locked (issue #238). No Node-version dependence — the app always runs on the bundled Node.

This is the practical advantage CodeGraph ships over every "install with npm" competitor. There is no node-gyp, no prebuild-install, no Windows ARM64 toolchain failure. The shell installer at install.sh just untars a per-platform release archive (codegraph-darwin-arm64.tar.gz, etc.) and symlinks one launcher script. Cross-compilation is free because there is nothing to compile.

The price is one hard exit: any user on Node 25.x is blocked with a custom banner explaining a V8 turboshaft WASM JIT zone allocator bug (linked back to issue #81). The bundled-runtime path is unaffected.

3.3 Extraction — 20 languages, every wrapper around tree-sitter justified

The 19 per-language extractors under src/extraction/languages/ follow one shape: a small LanguageExtractor object that names the tree-sitter node types corresponding to each NodeKind. The TypeScript one in src/extraction/languages/typescript.ts L4-50 is representative:

export const typescriptExtractor: LanguageExtractor = {
  functionTypes: ['function_declaration', 'arrow_function', 'function_expression'],
  classTypes: ['class_declaration', 'abstract_class_declaration'],
  methodTypes: ['method_definition', 'public_field_definition'],
  interfaceTypes: ['interface_declaration'],
  importTypes: ['import_statement'],
  callTypes: ['call_expression'],
  nameField: 'name', bodyField: 'body',
  resolveBody: (node, bodyField) => {
    // public_field_definition (arrow function class fields) nest the body inside
    // an arrow_function or function_expression child:
    //   public_field_definition → arrow_function → body (statement_block)
    // Also handles wrapper patterns like: field = withBatchedUpdates((e) => { ... })
    if (node.type === 'public_field_definition') { /* … */ }
    return null;
  },
  /* … */
};

The size distribution of the language extractors is the second tell that the project takes coverage seriously rather than just claiming it: Kotlin is 244 lines, Dart 195, C/C++ 163, Scala 143 (per wc -l). These are not five-line shims around a tree-sitter grammar; each one carries language-specific qualified-name logic, receiver-type extraction, and signature reconstruction (see the SEARCH_QUALITY_LOOP.md validation guide for the "GOOD vs BAD qualified_name" checklist a new language must pass).

Parsing happens in a dedicated worker thread that is recycled every 250 files (src/extraction/index.ts L41-49):

WASM linear memory can grow but NEVER shrink (WebAssembly spec limitation). The only way to reclaim tree-sitter's WASM heap is to destroy the entire V8 isolate by terminating the worker thread and spawning a fresh one. This interval balances memory usage against the cost of reloading grammars.

A PARSE_TIMEOUT_MS = 10_000 per file keeps a pathological input from freezing the whole index (L35-40). Files larger than 1 MB are skipped outright (L97-100) — minified bundles and vendored blobs blow the WASM heap with no useful symbols. The boring decisions are all in place.

The default-ignore set in src/extraction/index.ts L116-160 is one of those things that looks trivial until you scroll the list. ~40 directory names — node_modules, target, Pods, .next, .dart_tool, vcpkg_installed, __history (Delphi backup), bazel-*/ — curated from canonical github/gitignore templates plus a deliberate refusal to add generic names (packages, lib, src) that would hide real source. The comment is explicit: "First-party-prone or generic names are deliberately NOT listed, to avoid ever hiding real source." The default applies uniformly with or without git; the only opt-in is an explicit .gitignore negation (!vendor/).

File enumeration tries git ls-files --recurse-submodules first (so submodule-monorepos actually index — see issue #147), descends into embedded non-submodule repos one extra level (issue #193), and falls back to a filesystem walk in non-git projects. The Sync path uses git status --porcelain to detect changes by-name when possible (L289-300), so an incremental re-index after a one-file save is sub-second on a 10k-file repo.

3.4 Resolution — 21 framework resolvers + dynamic-dispatch synthesizers

The resolution layer is where CodeGraph spends most of its complexity budget. Tree-sitter gives you "method foo is called at file:line"; the user wants "method foo calls UserService.login defined in services/user.ts". Closing that gap takes two kinds of code.

The first is import resolution — turning import { X } from './foo' into an edge from this file's nodes to foo.ts's nodes — and the codebase handles tsconfig path aliases (@/, ~/), Cargo workspace members, Go modules, JVM fully-qualified names, and C/C++ include search paths from compile_commands.json. The C/C++ resolution in particular is recent (released in v0.9.6 per CHANGELOG entry [0.9.6]) and the changelog quantifies the effect on a known repo:

Measured on bitcoin-core (1,989 indexed files): C/C++ file→file imports edges 6,027 → 8,086 (+34%), false-positive call edges from std::move/std::swap etc. into similarly-named user methods −2,154 (−3.6% of C/C++ calls).

That's the rare changelog entry that gives you a count both ways — edges gained, false-positives removed.

The second is framework-aware resolution. src/resolution/frameworks/index.ts L31-69 registers 21 resolvers:

laraveldrupalexpressnestjsreactsveltevuedjangoflaskfastapirailsspringplaygorustaspnetswiftUIuikitvaporswift-objc-bridgereact-native-bridgeexpo-modulesfabric-view

Each one walks its language's AST looking for the framework's idioms — @RestController in Spring, path() / re_path() in Django, Route::get in Laravel, RCT_EXPORT_METHOD in React Native — and emits route nodes plus references edges. The Spring + MyBatis flow is the most ambitious; from the v0.9.6 changelog:

MyBatis XML mapper indexing + Java↔XML bridge. *.xml files containing <mapper namespace="..."> are now first-class: each <select|insert|update|delete id="X"> and <sql id="X"> becomes a method-shaped node qualified as <namespace>::<id>, and a new synthesizer (mybatis-java-xml) links the matching Java mapper interface method → its XML statement with a calls edge. […] Validated on macrozheng/mall-tiny: all 6 custom-SQL Java mapper methods reach their XML counterparts; trace(UmsRoleController.listResource, UmsResourceMapper::getResourceListByRoleId-xml) connects in 4 hops across controller → service-iface → impl → mapper-iface → XML.

The dynamic-dispatch synthesizers in src/resolution/callback-synthesizer.ts are the project's real differentiator. Static parsing breaks at every callback, observer, event emitter, or React setState. The synthesizer bridges them — each emitted edge tagged with both provenance:'heuristic' and the channel that wired it (synthesizedBy:'react-render', 'jsx-render', 'rn-event-channel', etc.):

// src/resolution/callback-synthesizer.ts L228-235
edges.push({
  source: stateUpdater.id, target: render.id, kind: 'calls',
  provenance: 'heuristic',
  metadata: {
    synthesizedBy: 'react-render', via: 'setState',
    registeredAt: `${render.filePath}:${render.startLine}`
  },
});

The principle is in CLAUDE.md L132: "Partial coverage is WORSE than none." Bridging one boundary but not the next reveals a hop the agent then drills + reads to finish; the worked Excalidraw example in the same file shows react-render alone raised file reads to 5–7, and only completing the flow (adding the JSX-child hop) dropped it to 0–1. That is the kind of nuance you only learn by running the A/B and watching where the agent gets stuck.

3.5 The MCP tool surface — 10 tools, sized for "stop reading"

The MCP server exposes exactly 10 tools, defined in src/mcp/tools.ts L357-570:

ToolPurpose
codegraph_searchQuick symbol search by name — locations only, no source.
codegraph_contextPRIMARY entry tool: composes search + node + callers + callees in one call for "how does X work" questions.
codegraph_callers / codegraph_calleesWalk the call graph one hop at a time.
codegraph_impactWhat's affected by changing this symbol, configurable depth.
codegraph_nodeOne symbol's details + its trail (what it calls, what calls it); optional source.
codegraph_exploreSource for several related symbols grouped by file, capped, with relationships.
codegraph_traceTHE marquee tool: shortest call path from from to to, each hop inlined with body.
codegraph_filesIndexed file tree, faster than filesystem glob.
codegraph_statusIndex health + statistics.

The interesting one is codegraph_trace, handled in src/mcp/tools.ts L1212-1326. It runs GraphTraverser.findPath along calls edges, caps at 7 hops because "a BFS shortest path longer than this on a dense call graph is almost always a spurious wander through unrelated code", and when no static path is found surfaces the start symbol's outgoing callees with a note about dynamic dispatch:

// src/mcp/tools.ts L1255-1264
'The direct chain most likely breaks at **dynamic dispatch** (a callback, descriptor, ' +
'metaclass, or attribute-as-callable) that static parsing cannot resolve into an edge. ' +
`Inspect \`${start.name}\` (${start.filePath}:${start.startLine}) with codegraph_node ` +
'(includeCode=true) — its body usually shows the dynamic call to follow next.',

When a path is found, every hop is inlined with its body (capped per hop), every dynamic-dispatch hop is annotated with its registration site, and the destination's own first-level callees are inlined too. The principle, from a comment on L1308-1311:

The "last mile": what the destination does next. Agents otherwise explore/Read for exactly this (e.g. renderStaticScene → _renderStaticScene → the canvas draw), so inlining the destination's callees is what actually stops the investigation — sufficiency, not a "don't explore" instruction.

The companion codegraph_explore carries an adaptive output budget sized to the project (src/mcp/tools.ts L80-188). A 300-file repo gets maxOutputChars=18,000, defaultMaxFiles=5, maxCharsPerFile=3,800; a 10k-file repo gets 35,000 / 12 / 7,000; a 25k+-file repo gets 38,000 / 14 / 7,000. The invariant is documented next to the code: "a larger tier must never get a smaller maxCharsPerFile than a smaller tier" — a regression they actually hit when an earlier <5000-tier value of 2,500 was below the <500-tier's 3,800, so on Excalidraw's 415 KB App.tsx one explore call returned <1% of the file and forced the agent into a Read.

That comment is the whole project distilled into eight lines. Local-first code intelligence is a measurement problem before it is a tooling problem, and the comments are written for whoever runs the next benchmark.

3.6 Auto-sync — the watcher that knows it's behind

The third-most-cited issue in the codebase (#276, #407, #449, #450) is the file watcher. It uses chokidar with the same buildDefaultIgnore filter the indexer uses, so the inotify set the kernel registers is exactly the set you index — the fix for #276 where a recursive fs.watch on a 10k-file monorepo exhausted the inotify budget.

Debounce is 2 seconds by default, clamped to [100 ms, 60 s] via CODEGRAPH_WATCH_DEBOUNCE_MS. The smart bit is the per-file staleness banner: in the brief window between an edit and the next sync, any MCP tool response that would reference a still-pending file prepends a ⚠️ banner naming the file and telling the agent to Read it directly (validated against Claude Code, where the agent literally says "Reading the file directly for the live content"); pending files not referenced by the response surface as a small footer instead. The agent gets an explicit signal either way.

3.7 The shared daemon — issue #411 and why proxies exist now

Pre-0.9, every agent that launched codegraph serve --mcp got its own process: its own SQLite handle, its own inotify set, its own grammar warm-up. On a developer's machine with Claude Code, Cursor, and Codex all open on the same repo that's three of everything. Issue #411 tracked the fix; the result is a three-process model documented at the top of src/mcp/index.ts L1-35:

Daemon

a detached background process (its own session/process group) that serves N proxies over a Unix-domain socket (Windows named pipe), sharing one CodeGraph + watcher + SQLite handle. Spawned on demand, reaped by client-refcount + idle timeout (default 300 s).

Proxy

what an MCP host actually talks to: a thin stdio↔socket pipe. Carries the #277 PPID watchdog (poll process.ppid every 5 s) so a SIGKILL'd host reaps its proxy promptly; the proxy's socket close decrements the daemon's refcount.

Direct

pre-#411 behavior, kept as a fallback when CODEGRAPH_NO_DAEMON=1, no .codegraph/ is reachable, or the daemon machinery fails.

The detached-daemon split is the fix for the review finding that the original in-process daemon (a) was the first host's child — so closing that terminal severed every other client — and (b) disabled the PPID watchdog, regressing #277.

There's a versioned hello-line handshake between proxy and daemon (src/mcp/daemon.ts L70-75): every connection emits { codegraph: <version>, pid, socketPath, protocol: 1 }. A 0.9.x proxy refuses to pipe through a 0.10.x daemon and falls back to direct mode. That is the kind of forward-compatible plumbing most v0.x projects skip.

3.8 Multi-agent installer — one extra agent is one new file

The installer at src/installer/ targets eight agents today: Claude Code, Cursor, Codex CLI, opencode, Hermes Agent, Gemini CLI, Antigravity IDE, Kiro. Each is one file under targets/ plus one line in targets/registry.ts, per the CLAUDE.md L72-83 architecture note. Adding a ninth (Continue, Zed, Windsurf) is "one new file in targets/ + one entry in registry.ts."

A few quality details:

  • opencode reads opencode.jsonc by default; edits are surgical via jsonc-parser so user comments and formatting survive install/re-install/uninstall round-trips.
  • Codex needs TOML; rather than add a dependency, the installer ships a 154-line hand-rolled serializer scoped to [mcp_servers.codegraph] (src/installer/targets/toml.ts). Sibling tables and [[array_of_tables]] are preserved verbatim.
  • Cursor's MCP working-directory quirk (launches MCP subprocesses with the wrong cwd, doesn't pass rootUri) is patched by the installer injecting --path into Cursor's MCP args — absolute path for local installs, ${workspaceFolder} for global. See the explicit "If you touch Cursor wiring, preserve this" warning in CLAUDE.md L85-87.
  • The installer is covered by 82 parameterized contract tests in __tests__/installer-targets.test.ts — install idempotency, sibling preservation, uninstall reverses install, byte-equal re-runs returning unchanged, partial-state recovery.

3.9 Tests — ~989 cases, real SQLite, no mocking

The test pyramid is unusual in two ways. First, the count: __tests__/ contains 48 *.test.ts files with ~989 individual it()/test() cases (and a separate, build-required __tests__/evaluation/ agent-eval harness run via npm run eval). Second, the kind: per CLAUDE.md L167:

Tests create temp dirs with fs.mkdtempSync and clean up in afterEach. They write real files and exercise real SQLite — there is no DB mocking.

You see the consequences in __tests__/security.test.ts (P0/P1 fixes: FileLock cross-process locking, path-traversal prevention via validatePathWithinRoot and validateProjectPath in src/utils.ts L40-100, MCP input validation, atomic writes), and in the explicit Windows-gated tests pattern documented in CLAUDE.md L169-171:

Behavior that differs by platform (path resolution, drive letters, SENSITIVE_PATHS, %APPDATA% config dirs, CRLF) must be gated, not assumed. Use it.runIf(process.platform === 'win32')(...) for Windows-only assertions and it.runIf(process.platform !== 'win32')(...) for POSIX-only ones — e.g. /etc is sensitive on POSIX but resolves to C:\etc (non-existent) on Windows, so an ungated /etc assertion fails on Windows.

The Windows validation is done on a real Parallels VM over SSH (CLAUDE.md L185-199) and the Linux validation in Docker with --init (L177-184) — the latter because without a zombie-reaping PID 1 a SIGKILL'd process lingers and the daemon lifecycle tests false-pass. This is the kind of platform-test hygiene that single-maintainer v0.x projects almost never bother with.

The vitest config carries one revealing exception (vitest.config.ts L14-24): it sets CODEGRAPH_ALLOW_UNSAFE_NODE=1 in the test environment so the Node 25.x hard-block doesn't fire for contributors who happen to be on a Node 25 dev machine. The CLI's user-facing guard stays intact. That's a thoughtful response to PR feedback (#478) that recognized "block users but unblock tests".

Two things to notice. First, the gains scale with codebase size and complexity. On a 110-file Gin repo native search is already cheap; on a 10k-file VS Code the no-CodeGraph arm fans out into 55 tool calls vs CodeGraph's 8. Second, the maintainer's own docs/benchmarks/codegraph-ab-matrix.md — a 37-cell matrix across every language × repo-size — is more honest than the README. The matrix's "Headline" paragraph admits:

Cost stays roughly flat — marginally higher on the with-arm here (summed across the 37 cells: with $15.4 vs without $13.8). The win is fewer tool calls (189 vs 321, −41%) + lower wall-clock (mean 38s vs 48s), which is the design target.

That nuance — "cost only flips on multi-turn investigations where the without-arm's accumulated context balloons" — is what most v0.x README marketing leaves out. CodeGraph's leaves it in (in the benchmark doc, anyway).

4.2 The CLI surface — the bits the README hides

The CLI in src/bin/codegraph.ts (1,800+ lines) ships sixteen subcommands. Most are obvious; two are worth flagging.

codegraph affected (src/bin/codegraph.ts L1494-1620) takes a list of changed source files (or --stdin, e.g. from git diff --name-only) and BFS-walks the file→file imports graph to find which test files transitively depend on them. Default test patterns are .spec., .test., /__tests__/, /tests?/, /e2e/, /spec/; custom globs are accepted. The expected use is a pre-commit or CI hook:

AFFECTED=$(git diff --name-only HEAD | codegraph affected --stdin --quiet)
[ -n "$AFFECTED" ] && npx vitest run $AFFECTED

This is a real, valuable use of the index outside the AI-agent loop, and most competitors don't ship it. Limitations are worth knowing: it's imports-only (so dynamic-dispatch tests are missed exactly the way trace is) and default depth is 5 — past that you get nothing.

codegraph context "<task>" (command at L1096-1141) takes a natural-language task description, runs the same ContextBuilder the MCP tool uses, and prints a markdown or JSON context bundle to stdout. The intended use is paste-into-prompt, but the same call from a script lets you build agent-free pipelines. Underdocumented; underrated.

4.3 What the README doesn't tell you

A fair deep-dive has to name the things the README omits. The list is short, but worth knowing.

The maintainer's own notes admit tool-choice is the bottleneck, not retrieval quality. CLAUDE.md L99-111 is the most honest section in any v0.x README I've read this year:

CodeGraph's only channels to influence the agent are low-salience: the MCP initialize instructions (server-instructions.ts) and the tool descriptions. Changing them does not reliably move the agent's tool choice or query style — validated: trace-first steering ported into the server-instructions + tool descriptions (3 wording variants) never reproduced what a CLI --append-system-prompt achieved, and regressed wall-clock vs baseline.

Translation: getting the agent to actually use codegraph_trace instead of falling back to Read is an unsolved problem outside what the MCP server can control. The Excalidraw worked example shows the trace-first steering improved best-run to 3 codegraph / 0 reads / 0 greps / 51s vs the unsteered 3–10 codegraph / 0–2 reads / 0–4 greps / 64–112s — but the steering had to land at the CLI prompt level, not in the MCP server. The MCP server alone can't make Claude Code call trace; that depends on the host model's tool-choice behavior, which improves on its own as models get better at tool use but is not the maintainer's lever.

Bus factor is real. git shortlog -sn against the 377-commit history: Colby McHenry 317, Olaf Monien 16, andreinknv 7, everyone else ≤ 4. That's 92% of commits by one author over a 4-month project history. Compare to Mem0's 200+ contributors over three years; CodeGraph is one engineer's vision shipped fast.

Release cadence is "still finding shape". 12 releases in the public history (v0.7.6 → v0.9.6), with v0.9.0 (2026-05-21) → v0.9.6 (2026-05-27) = 7 releases in 7 days. That's healthy for a v0.x project, but if your CI pins a specific version, expect to bump it weekly.

Local-only is a feature, but it also caps the ceiling. No embeddings, no semantic search, no LLM-derived nodes. Search is FTS5 + name/qualified-name LIKE matching with a small field-qualified DSL (kind:function lang:python authenticate — src/search/query-parser.ts L1-30). That's fine for "how does X reach Y" but the project quietly disclaims semantic gaps: the mcp_context tool description ends with "NOTE: provides CODE context, not product requirements; for new features still clarify UX/edge cases with the user" (src/mcp/tools.ts L385).

SQLite WAL isn't free everywhere. README L564-568:

codegraph status shows Journal: other than wal — WAL couldn't be enabled on this filesystem (common on network shares and WSL2 /mnt), so reads can block on writes. Move the project (with its .codegraph/ folder) onto a local disk.

If your repo lives on WSL2's /mnt/c/... mount or a corporate SMB share, the daemon model can stall. Known issue, documented, but a real ceiling on the "100% local" claim.

203 open issues against 31k stars. That ratio is healthy for an active project; the open ones I sampled (#525 watcher behavior, #524 ReScript request, #517 watcher fd leak on nested .git, #516 non-ASCII keyword extraction, #518 sub-directory invocation) are mostly real edge cases on real codebases rather than fundamental design problems. The triage discipline visible in the closed PRs (issue numbers in every commit message) is encouraging.

5 · Verdict — who should install this today

Two strong yeses, two reasonable nos, four things to watch.

5.1 Two strong yeses

Solo developers on a non-trivial repo

If you're using Claude Code, Cursor, or Codex on any repo over ~500 files, the default agent fans out 20–80 tool calls for any architecture question; CodeGraph drops that to 3–10. On a TypeScript or Rust monorepo the savings are immediate and obvious.

curl -fsSL https://raw.githubusercontent.com/colbymchenry/codegraph/main/install.sh | sh && codegraph init -i

is two minutes of work. The license is MIT, the data never leaves your machine, the worst case is you uninstall it.

Tooling teams evaluating AI-native code intelligence

The pattern of "tree-sitter graph + adaptive output budgets + dynamic-dispatch synthesizers + provenance-tagged edges" is the right shape, and CodeGraph's source — particularly src/mcp/tools.ts and callback-synthesizer.ts — is some of the most clearly-commented MCP-server code in the open-source ecosystem.

MIT license means you can lift the patterns.

5.2 Two reasonable nos

Teams wanting semantic search

"Find me code that does X" where X is a concept, not a name — CodeGraph doesn't do that. It's FTS5 + name matching + structural edges.

If you want vector retrieval over code, use Aider's repo-map, Continue's @codebase, Cursor's built-in embeddings index, or hosted offerings like Sourcegraph Cody.

Multi-developer / multi-machine indexing

The whole architecture is "one daemon per user per machine." There's no central index server, no shared .codegraph/ mode, no team-scope.

A team of ten engineers indexing the same monorepo ten times is the design — fine on individual laptops, wasteful on shared dev infra.

5.3 What to watch out for

  • Pin your version. npx @colbymchenry/codegraph@0.9.6 rather than npx @colbymchenry/codegraph; the release cadence guarantees you'll hit a transient bug if you don't.
  • Check codegraph status after install. If Journal: reports anything other than wal, move your .codegraph/ to a local disk.
  • Disable on tiny repos. The Gin (110-file) cell in the 37-cell A/B shows the MCP overhead doesn't amortize. codegraph uninit is one command.
  • Don't expect the agent to always pick codegraph_trace. The maintainer's own notes say tool-choice steering is fragile; on some sessions the model will still fan out into Reads. That's an LLM limitation, not a CodeGraph one, but it's where the gain variance comes from.

Honest read: if you want a standalone code-intelligence MCP server for Claude Code, the choice in 2026 is CodeGraph vs Serena.
