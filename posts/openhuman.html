Open-source agent harness

OpenHuman

A Tauri + Rust desktop agent that puts the memory of your life on your own machine, plugs into 118+ services with one-click OAuth, and dispatches each turn to the right model under one subscription. This is what's actually inside, what it's good at, and where it sits versus ChatGPT, Claude, Copilot, Comet and Open Interpreter.

What is OpenHuman?

Marketed as a "personal AI super intelligence — private, simple, extremely powerful." Underneath that line is a fairly disciplined architectural bet: most assistants are stateless chat windows; OpenHuman is built around a persistent, local-first memory of your life and a router that picks the right model per task.

Memory you own

Every email, message, doc, repo and calendar event is canonicalised into Markdown chunks, scored, and folded into hierarchical summary trees stored in SQLite on your own laptop. Mirrored to an Obsidian-compatible vault you can read and edit.

An agent that fetches itself

118+ OAuth integrations. Every 20 minutes the core walks each connection and pulls fresh data into the Memory Tree. By morning it already has yesterday's emails, PRs and Slack threads in context.

One subscription, many models

A built-in router sends each turn to the right model — reasoning, fast, vision, summarise, code — across Anthropic, OpenAI, Google, Groq and an optional local Ollama backend. No BYO-key juggling.

Architecture

A monorepo with one Rust crate (openhuman_core) doing the heavy lifting and one TypeScript app (openhuman-app) doing the UI. Tauri 2 ties them together; Chromium Embedded Framework is bolted on for browser/computer control.

OpenHuman conceptual architecture: desktop mascot, Memory Tree of markdown documents, and 118+ integration streams

A desktop with a mascot, a Memory Tree of Markdown leaves growing behind it, and streams of integration data flowing in. The technical layout is below.

Why Tauri + Rust, not Electron

Per-skill context costs ~1–2 MB on the embedded QuickJS runtime instead of a full Chromium renderer. Cold start is sub-500 ms. Tokio drives all I/O — WebSockets, HTTP, file work — as non-blocking tasks on a small thread pool. TLS goes through rustls, not OpenSSL. The trade-off: more platform work and no mobile/web targets today.

Dual-socket realtime

A Rust-native WebSocket client (tokio-tungstenite + rustls) on desktop survives app backgrounding and runs independently of the WebView; a JS Socket.io client is reserved for the experimental web build. Telegram, Discord and triggers ride the persistent socket.

Skills as sandboxed JS

Skills are JS modules executed inside QuickJS with enforced resource limits and a typed bridge to Rust APIs. They install from the Skills tab, react to triggers, run on cron, and write into the same Memory Tree as everything else.

Browser & computer control via CEF

An embedded Chromium surface lets the agent open URLs, take screenshots, click, type and drag. Movement uses human-like trajectories (anti-bot-detection). Useful for sites with no API and for driving local apps from chat.

The five pillars

Strip away the marketing and OpenHuman is doing five things that most "AI assistant" apps don't.

1 · Memory Tree

Local SQLite + Markdown vault. Three trees — per-source, per-topic, daily global — built from scored chunks. Embeddings live inside but the structure on top is what makes the memory feel like a brain instead of a bag of fragments.

SQLiteMarkdownObsidian-compatibleLocal-first

2 · Auto-fetch

Every 20 minutes a worker walks each active connection and pulls fresh data into the tree. By morning the agent already has overnight Slack, last night's GitHub events, and today's calendar. No "give it a few weeks" cold-start penalty.

20 min loop118+ providersComposio-backed

3 · TokenJuice

A three-layer JSON rule overlay (built-in / user / project) compacts every tool result before it touches an LLM. A git status, a 600-message email thread, a kubectl dump — all reduced before paying tokens. Claim: ingesting six months of email costs single-digit dollars instead of hundreds.

Builtin → User → ProjectJSON rulesHot-reloaded

4 · Model router

Each chat call passes either a concrete model or a hint — hint:reasoning, hint:fast, hint:vision, hint:summarize, hint:code. The router resolves it to a (provider, model) at runtime. The agent loop emits hints; the user never picks. One subscription, many providers.

AnthropicOpenAIGoogleGroqOllama (local)

5 · Mascot · voice · meeting agent

A Remotion-rendered desktop face with viseme-driven lip-sync, hosted TTS, STT, and a Google Meet bot that joins as a real participant — listens, transcribes into the Memory Tree, and speaks back into the call. A "subconscious" loop keeps it thinking while you're idle; a "dreaming" pass consolidates the day when you're away.

Inside the Memory Tree

This is the load-bearing piece. Most "AI memory" features are a sticky note bolted to a chat. OpenHuman's is a deterministic, bucket-sealed pipeline.

Source trees

Per-source rolling buffer (L0) that seals into L1 → L2 → … as it fills. One per Gmail label, one per Slack channel, one per uploaded document. Gives you provenance: "what did the Stripe webhook say last Tuesday at 3pm?"

Topic trees

Per-entity summaries, materialised lazily by hotness. The more an entity (person, project, ticker, repo) shows up, the more aggressively its tree is rebuilt. Gives you "what's the latest on this person?"

Global tree

One daily UTC digest across everything ingested that day. Gives you "what happened today?" Built by a scheduled background pass, not at query time.

Read the OpenHuman column top to bottom and the design intent is clear: OpenHuman is making the bets nobody else is making — open-source license, three-tree local memory, batteries-included integrations, a structural token-compression layer. The trade-off is everywhere else: it is younger, less polished, and depends on the OpenHuman backend to broker LLM calls and OAuth.

Honest take

No product is all upside. Here's what looks genuinely strong and where the cracks are likely to show.

What's genuinely strong

  • The memory architecture. Trees + provenance + hotness is closer to how a brain remembers than any vector blob. The fact that the vault is plain Markdown means you keep your data even if the company disappears.
  • TokenJuice as a first-class layer. Putting compression at the tool/result boundary — not as a model trick — is the right place. It's what makes 20-minute auto-fetch sustainable.
  • Hint-driven routing. "The task picks the model, not the user" is a real cost-and-quality win once the agent is doing dozens of internal calls per turn.
  • Tauri + Rust + QuickJS. ~1–2 MB per skill versus ~150 MB per Electron renderer is not a vanity number — it's why the subconscious loop and the meeting bot can run continuously without melting the laptop.
  • Genuinely batteries-included. Voice, search, scraper, coder kit, browser, computer, cron, sub-agents and 118 integrations on day one is rare in open-source land.

Where it could bite

  • Single backend dependency. Unless you flip on Local AI (Ollama) for everything, every LLM call, every search, every TTS stream goes through the OpenHuman backend. That's a single vendor and a single trust boundary.
  • Composio-backed connectors. The 118+ figure leans on a third party. If Composio changes terms or pricing, the integration story changes with it.
  • Desktop-only, no mobile. The architecture doc is explicit: Android, iOS and a standalone web client are out of scope today.
  • GPL-3.0 means no closed-source forks. Good for the commons; a non-starter for some commercial integrations.
  • "Early beta" is real. Version 0.53.x, active development, the README warns "expect rough edges." The headline features (Memory Tree, mascot, Meet bot) are big surface areas to harden.
  • Always-on auto-fetch is a privacy posture choice. Even with local-first storage, syncing every 20 minutes from Gmail / Slack / Calendar is a meaningful trust ask. The privacy boundary is documented but worth reading before turning everything on.

So who is it actually for?

Buy in if you are…

  • An individual power user who wants your assistant to know your stack — Gmail, Slack, GitHub, Notion, Calendar — without writing glue code.
  • A developer who likes the idea of an agent that can debug code, read a repo, draft a PR, and take notes from a Google Meet — all from one app.
  • Someone who cares about owning the data: a Markdown vault on disk you can grep beats a server-side memory blob you can't see.
  • An OSS-friendly tinkerer happy to live near the bleeding edge.

Pass if you are…

  • An enterprise buyer needing tenant-level controls, SOC2 paper, an SLA, and a "no third party in the loop" story today.
  • A mobile-first user — there is no iOS or Android client.
  • Someone who needs the absolute frontier model on every turn — the router will sometimes pick fast/cheap models for cost reasons.
  • Allergic to GPL-3.0 in your stack.

My one-liner

My one-liner

OpenHuman is the most architecturally interesting open-source agent harness shipping in 2026. It treats memory, integrations, token compression and model routing as first-class system components rather than features bolted on later. Whether it survives contact with users at scale is the open question — but the bets it's making are the right ones.
