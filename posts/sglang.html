SGLang

The serving framework that treats prompt structure as a systems problem, not just an API shape. SGLang's backend — the SGLang Runtime, or SRT — is built around RadixAttention: automatic cross-request KV-cache reuse through a radix tree of token prefixes. The front end adds a Python DSL for multi-call generation programs, while the backend has grown into a full OpenAI-compatible LLM and vision-LLM server with structured decoding, speculative decoding, quantization, and distributed execution. The speed story is real, but so is the churn: SGLang is young, aggressive, and in direct benchmark leapfrog mode with vLLM.

In short

SGLang is what happens when prefix caching, structured generation, and a programmable prompting language are co-designed instead of bolted together.

What it is

A high-performance serving framework for large language models and multimodal models. It exposes OpenAI-compatible endpoints through python -m sglang.launch_server, an offline engine, and native APIs, while the engine core lives under python/sglang/srt/. The headline backend mechanism is RadixAttention: finished and in-flight prompts are retained in a radix tree so later requests can reuse any shared token prefix, not just an exact whole prompt.

Why it's interesting

Most inference engines now cache prefixes; SGLang made that cache the organizing abstraction. The runtime pairs RadixAttention with continuous batching, paged attention, overlap scheduling, tensor / pipeline / expert / data parallelism, speculative decoding, quantization, and grammar-constrained output. The front end is unusual too: a Python DSL with @function, gen(), choices, fork, control flow, streaming, batching, and multimodal inputs.

Biggest concerns

It is younger and faster-moving than some peers, with ~3.6K open issues and APIs that change as the engine expands. There is heavy overlap with vLLM; the projects borrow ideas and leapfrog each other on benchmarks. The DSL is powerful but has a smaller ecosystem than OpenAI-compatible HTTP alone, and every speed claim is workload-dependent: shared prefixes, structured output, long context, model family, and hardware all matter.

Verdict

SGLang is the most interesting vLLM peer because it is not merely another server wrapper: RadixAttention gives it a coherent cache model, and the DSL makes multi-call generation programs first-class. Reach for it when your workload has shared prefixes, structured outputs, long-context agents, VLM traffic, or RL rollout pressure. But pin versions, read release notes, and benchmark your exact model / quant / GPU tuple. The best SGLang numbers come from workloads that line up with its cache and scheduler assumptions; the worst surprises come from treating benchmark plots as universal constants.

First — a generation program, not a prompt

SGLang's name is the tell: it is a Structured Generation Language with a runtime under it, not just a server. The smallest program that shows why both halves exist:

import sglang as sgl

@sgl.function
def triage(s, ticket):
    s += sgl.system("You are a Spark support agent. Tools: [docs, logs].")
    s += sgl.user(ticket)
    s += sgl.assistant(sgl.gen("answer", max_tokens=256))

sgl.set_default_backend(sgl.RuntimeEndpoint("http://localhost:30000"))
state = triage.run(ticket="Why is my join spilling to disk?")
print(state["answer"])

The function reads like ordinary Python — system / user / assistant turns and a named gen field you read back as state["answer"] — but every call funnels into a runtime (SRT) you launch separately:

$ python -m sglang.launch_server \
      --model-path Qwen/Qwen2.5-7B-Instruct \
      --tp 2 --grammar-backend xgrammar
# server is up on http://0.0.0.0:30000  (OpenAI-compatible)

That server also speaks the OpenAI API, so you never have to touch the DSL if you would rather not. Ask it for JSON and it will guarantee the shape, not just request it politely:

$ curl localhost:30000/v1/chat/completions -d '{
    "model": "Qwen/Qwen2.5-7B-Instruct",
    "messages": [{"role": "user", "content": "Classify: disk spill on shuffle join"}],
    "response_format": {"type": "json_schema", "json_schema": {
        "name": "triage",
        "schema": {"type": "object",
          "properties": {"severity": {"enum": ["low", "high"]}, "area": {"type": "string"}},
          "required": ["severity", "area"]}}}
  }'

Two front doors (DSL and OpenAI API), one runtime, and underneath both the same two bets: reuse KV cache across every call that shares a prefix, and fold constrained output into the decode loop. The rest of this piece is those two bets.

1 · Situation — LLM programs stopped being single prompts

The original SGLang problem statement is still the right one: real LLM applications are not one prompt, one generation, done. Agents branch. RAG pipelines reuse the same documents. Few-shot evaluation repeats the same examples across many questions. Multi-turn chat drags a growing history through every request. Structured extraction alternates between free generation and rigid syntax. Vision-language models add image tokens that are expensive to encode and easy to repeat.

That creates a systems opportunity. If many requests share substrings of their prompt, recomputing the KV cache for those substrings is waste. Prefix caching catches the simplest case — identical leading prompts — but complex programs create a richer set of partial overlaps: a shared system prompt, then different user questions; a shared few-shot block, then a batch of examples; a chat branch that diverges and later reuses the first turns; a tree-of-thought search that fans out from a common path.

SGLang's thesis is that the runtime should discover and exploit that structure automatically. The front end can express complex generation programs, but the backend should still accept full prompts and find reusable prefixes without hand-authored cache keys. That is the gap RadixAttention was built to fill.

2 · Task — make structured generation fast without hiding the engine

The task is broader than "serve tokens faster." SGLang wants to make the common hard cases fast: multi-call generation, constrained JSON, parallel branches, multimodal prompts, and high-throughput serving across GPU clusters.

That splits into two layers. The frontend language should let users write Pythonic generation programs with control flow and parallelism instead of building fragile prompt strings. The backend should run those programs, or plain OpenAI-compatible requests, while reusing KV cache across calls, scheduling continuously, and keeping the GPU fed. The pieces that matter most:

  1. RadixAttention — a radix-tree KV cache for arbitrary shared token prefixes with eviction.
  2. SRT — the runtime under python/sglang/srt/, including scheduler, model executor, memory cache, grammar manager, and distributed paths.
  3. Structured output — JSON schema, regex, EBNF, and structural tags through XGrammar, Outlines, or llguidance.
  4. Production serving knobs — --tp, --dp, --mem-fraction-static, --chunked-prefill-size, --grammar-backend, quantization, speculative decoding, and hardware-specific attention backends.

3 · Action — what SGLang actually built

Six lenses: RadixAttention, the Python DSL, structured decoding, the scheduler, parallel execution, and the quantization / model matrix.

3.1 RadixAttention — prefix reuse as a radix tree

RadixAttention stores token sequences and their KV-cache indices in a radix tree. In the current tree implementation, python/sglang/srt/mem_cache/radix_cache.py defines RadixKey, TreeNode, RadixCache.match_prefix(), insert(), cache_finished_req(), and cache_unfinished_req(). A request is represented by token ids plus an optional extra_key namespace, which keeps LoRA adapters or cache-salted requests from sharing state accidentally. Matching can be page-aligned through page_size, and the cache tracks access metadata for eviction.

The key difference from ordinary prefix caching is granularity. A radix tree can split edges when two sequences share only part of a stored segment, so the cache can reuse arbitrary common sub-prefixes that start at the root. In the original SGLang write-up, this covered chat histories, few-shot examples, self-consistency, and tree-of-thought branches. In modern serving terms, it is exactly the pattern that shows up in agent frameworks: a fixed instruction scaffold, a repeated tool schema, shared retrieved documents, then many divergent continuations.

Eviction is no longer a footnote. Current server arguments list RADIX_EVICTION_POLICY_CHOICES as lru, lfu, slru, and priority; the original design used LRU leaf eviction because GPU memory is finite and cached generations cannot grow without bound. The operational point is simple: RadixAttention is excellent when traffic clusters around shared prefixes, but cache hit rate is a workload property, not a constant.

A picture makes the "arbitrary shared sub-prefix" claim concrete. Three agent requests share a system prompt and tool schema; two of them also share a retrieved Spark-docs block; the questions then diverge. RadixAttention stores that as one tree and prefills only the unique tail of each request:

The payoff is mechanical: if the shared path is 2,000 tokens and a new request adds 30, the engine prefills 30 tokens, not 2,030. Where ordinary prefix caching needs the whole leading prompt to match byte-for-byte, the radix tree reuses any segment that starts at the root — which is why chat histories, few-shot blocks, and tree-of-thought branches all hit.

3.2 The front-end DSL — Python control flow around generation

SGLang's name comes from the Structured Generation Language, not from the server alone. The DSL embeds generation in Python: decorate a function with @function, append system(), user(), and assistant() messages to a state, call gen("name") to produce named fields, and read them back with s["name"]. Normal Python if statements choose branches; fork creates parallel prompt copies; run_batch batches inputs; streaming returns partial text.

This is not just syntactic sugar. The frontend can express intra-program parallelism that the backend can batch, while RadixAttention can reuse the shared parts across those branches. The DSL can run against a local SGLang runtime or remote backends such as OpenAI-compatible APIs, which keeps it useful as a programming model even when the local runtime is not in play.

The caveat is ecosystem size. HTTP clients, OpenAI SDKs, LangChain-style adapters, and inference gateways are everywhere; a specialized generation DSL asks users to learn a smaller language. The upside is precision for complex workflows; the downside is that many teams will use only the server and never touch the frontend.

@sgl.function
def compare(s, topic):
    s += sgl.user(f"Evaluate {topic}.")
    pro, con = s.fork(2)              # two branches; shared prefix served from cache
    pro[0] += sgl.user("Give only the strongest PRO.")
    con[0] += sgl.user("Give only the strongest CON.")
    pro[0] += sgl.assistant(sgl.gen("pro", max_tokens=64))
    con[0] += sgl.assistant(sgl.gen("con", max_tokens=64))
    return pro[0]["pro"], con[0]["con"]

The fork splits the program into two branches the backend can batch together, while RadixAttention serves their shared prefix from one copy of the KV cache. This is the case the DSL expresses cleanly and a hand-built prompt string cannot: intra-program parallelism the runtime can actually exploit.

3.3 Structured decoding — JSON, regex, EBNF, and structural tags

SGLang has been unusually serious about constrained output. Current docs expose json_schema, regex, ebnf, and structural_tag through both OpenAI-compatible responses and native /generate sampling params. The server can use --grammar-backend xgrammar, outlines, llguidance, or none; server_args.py makes XGrammar one of the first-class grammar backend choices, and grammar_manager.py compiles and caches grammar objects per request.

The historical mechanism was jump-forward decoding with a compressed finite state machine. Instead of sampling one token for every deterministic syntax transition, SGLang identifies singular FSM paths and extends through them, then resumes model sampling at the next branch. That was reported as up to 2× lower latency and 2.5× higher throughput for JSON-style tasks in 2024, and the v0.4 release moved further by integrating XGrammar, claiming up to 10× faster JSON decoding than other open-source paths for some workloads.

Those numbers are not universal, but the architecture is important. Structured output is not treated as a slow validation layer after generation. It is part of the decoding loop, and because jump-forward can terminate and enqueue new work while RadixAttention reuses the previous KV cache, the cache design and grammar design reinforce each other.

# native endpoint: constrain output with a regex (compiled to an FSM once, then cached)
$ curl localhost:30000/generate -d '{
    "text": "Ticket #4821 severity:",
    "sampling_params": {"regex": "(low|medium|high)", "max_new_tokens": 4}
  }'
# -> {"text": " high"}     no token outside the grammar can be sampled

Because the grammar compiles to a finite-state machine, the deterministic stretches — the quotes, braces, and field names of a JSON object — are jump-forwarded rather than sampled one token at a time, so the model spends forward passes only on the genuinely free fields. Constraint stops being a tax on throughput and becomes a shortcut.

3.4 Scheduler and batching — overlap the CPU with the GPU

SRT's scheduler is not a small queue in front of PyTorch. python/sglang/srt/managers/scheduler.py owns tensor-parallel workers, radix cache interactions, grammar queues, LoRA loading, speculative paths, disaggregation, multimodal processing, metrics, sessions, and distributed rank state. Continuous batching is table stakes, but SGLang's v0.4 claim was a near zero-overhead batch scheduler: run the CPU scheduler one batch ahead and prepare metadata while the GPU executes the current batch.

The practical knob is --disable-overlap-schedule, because overlap scheduling is on by default for supported configurations. This matters most when CPU-side work — prefix matching, memory allocation, grammar preprocessing, request accounting — starts to show up in Nsight as gaps between GPU kernels. The implementation uses future tokens, CUDA events, and careful dependency resolution to hide that overhead. It is also exactly the kind of optimization that can break at the edges, which is why release notes frequently mention scheduler overhead, async value passing, CUDA graph coverage, and backend-specific fixes.

3.5 Parallelism, disaggregation, and scale-out

SGLang now spans much more than single-node serving. Launch docs expose --tp for tensor parallelism, --dp through the router path for data parallelism, multi-node --nnodes, and specialized features such as expert parallelism, data-parallel attention, context parallelism, and prefill-decode / encoder-prefill-decode disaggregation. The README lists NVIDIA, AMD, Intel Xeon CPU, Google TPU, Ascend NPU, and Apple Metal paths; the repository language mix confirms this breadth: mostly Python, with significant MDX docs, Rust, CUDA, C++, shell, HIP, Metal, and CMake.

The router story is worth separating. The v0.4 cache-aware load balancer maintained an approximate radix tree of worker cache states so traffic could be routed toward the worker with the highest expected prefix hit rate. That is a direct consequence of making prefix cache state central: once cache hits matter this much, round-robin routing becomes wasteful. The router is Rust-heavy and packaged separately as sglang-router.

At the frontier-model end, release notes now mention Spec V2, DFlash, MTP, DeepSeek-specific MLA and sparse kernels, HiCache, heterogeneous CPU + GPU EPD disaggregation, and AMD MoRI. That is impressive, but it also means the project is carrying many specialized systems at once.

3.6 Quantization, models, and the support matrix

The quantization page is long because the matrix is real: FP8, MXFP4, MXFP8, blockwise INT8, W8A8 INT8/FP8, AWQ, GPTQ, compressed-tensors, Quark, auto-round, Marlin variants, GGUF, ModelOpt FP8/FP4, NVFP4 online, Petit NVFP4, bitsandbytes, torchao, and Ascend-specific paths. server_args.py mirrors that list in QUANTIZATION_CHOICES, while attention backends include Triton, FA3/FA4, FlashInfer, FlashMLA, TensorRT-LLM paths, Aiter, Wave, Intel AMX/XPU, Ascend, and more.

This is a strength if you are serving DeepSeek, Qwen, Llama, Kimi, GLM, Gemma, Mistral, VLMs, embeddings, reward models, LoRA adapters, or diffusion workloads on mixed infrastructure. It is also the reason operational validation matters. A model can work in BF16 on H100 and fail in FP4 on Blackwell with one attention backend; a VLM path can be correct while a structured-output path waits on grammar preprocessing; an AMD kernel can lag an NVIDIA path or vice versa. The feature breadth is not free.

The central result is not just higher throughput. It is a more cache-aware way to think about LLM serving. If your application repeats context — system prompts, tool schemas, documents, chat history, few-shot examples — SGLang has a first-class mechanism for turning that repetition into saved prefill work. If your traffic is mostly unrelated short prompts at low concurrency, the advantage narrows and the operational complexity may not pay for itself.

5 · Takeaways — how to read SGLang in 2026

Five calibrated conclusions for an engineer choosing a serving stack.

What's genuinely strong

  • RadixAttention is a clean abstraction. Prefix reuse becomes a tree operation instead of a pile of manual cache keys.
  • Structured output is a first-class path. XGrammar, regex, EBNF, JSON schema, and structural tags are integrated with serving rather than left to client-side retries.
  • The runtime is ambitious. Overlap scheduling, speculative decoding, disaggregation, HiCache, and MoE-specific kernels show a project chasing real bottlenecks.
  • Adoption appears broad. The project reports trillions of tokens per day, 400K+ GPUs, and major labs / clouds / infrastructure companies using it.

What to budget for

  • Version churn. v0.5.13 is still pre-1.0; pin versions and expect flags and defaults to move.
  • Benchmark locality. SGLang vs vLLM flips depending on shared prefix rate, structured decoding, model, quantization, GPU, and concurrency.
  • The support matrix. The repo spans CUDA, HIP, Rust router code, TPUs, NPUs, Metal, diffusion, VLMs, and RL rollout paths; each path ages differently.

Use it when structure is the workload

SGLang is strongest when the application has repeated context or generated structure: agents, RAG, multi-turn chat, self-consistency, JSON extraction, VLM extraction, RL rollouts, and shared-tool prompts.

Do not confuse the DSL with the server

You can use SGLang as a plain OpenAI-compatible server and ignore the frontend language. The DSL is a differentiator for complex programs, but the backend stands on its own.

Treat speed claims as hypotheses

The 5×, 10×, 1.9×, and 25×-style numbers in SGLang material are workload-specific. They are useful signals for where the project is optimized, not promises for your deployment.
