Delta Lake

The lakehouse format that made object storage feel less like a pile of files and more like a database table. Delta Lake wraps Parquet data with the transaction log: ordered JSON commits, periodic Parquet checkpoints, protocol versions, and enough metadata to reconstruct any snapshot. It is still Spark-native at the center, but delta-rs and Delta Kernel are turning the format into a shared substrate for non-Spark engines. The result is practical ACID on S3/GCS/ADLS, with real operational bills: commit coordination, small-file cleanup, protocol compatibility, and an Iceberg interoperability story that is useful but not free.

In short

A transactional table format for Parquet data that moves database semantics into the lake without moving the data out of the lake.

What it is

An Apache-2.0 open-source storage framework for building lakehouse tables over object storage and distributed filesystems. A Delta table is ordinary Parquet data plus a _delta_log/ directory containing newline-delimited JSON commit files, Parquet checkpoint files, and protocol metadata. The core repo is Scala/Java and Spark-native; Python ships through delta-spark; Rust/Python non-Spark work lives in delta-rs; connector authors increasingly target Delta Kernel.

Why it's interesting

It makes immutable files behave like a mutable table. Writers atomically add and remove files in the log, readers choose a consistent snapshot, and maintenance commands compact, cluster, vacuum, and rewrite layout without changing the logical table. Time travel, schema enforcement, MERGE, streaming, deletion vectors, and UniForm are all expressions of the same idea: the metadata layer is the table.

Biggest concerns

The center of gravity is still Spark and Databricks-shaped. Non-Spark support is improving through delta-rs and Kernel, but feature parity is uneven. Object-store commits are not magic; multi-cluster S3 writes need a safe commit coordinator or catalog-managed protocol. Small files require regular OPTIMIZE and VACUUM. Protocol bumps and table features can strand older readers.

Verdict

Delta Lake is the strongest choice when Spark is already your write path and you want ACID, DML, streaming, and mature operational commands without leaving Parquet. Iceberg has the cleaner multi-engine perception and a broader neutral-catalog story; Delta has a very battle-tested Spark implementation and a fast-moving interoperability push through Kernel and UniForm. The honest answer is not "Delta beats Iceberg" or the reverse. If your writes are Spark-heavy, Delta is pragmatic. If your organization is engine-diverse from day one, test Iceberg and Delta against the exact features you need, especially row deletes, catalog integration, and protocol compatibility.

1 · Situation — data lakes had no table contract

Classic data lakes were cheap, scalable, and operationally loose. They stored Parquet, ORC, or CSV files in folders on S3, ADLS, GCS, or HDFS, then asked query engines to infer table state from directory listings and partition paths. That worked for append-only logs. It fell apart as soon as users expected database behavior.

A failed job could leave half-written output beside valid data. A reader could list files while a writer was replacing a partition and see a state that never existed logically. Deletes and updates meant rewriting data files and hoping every engine agreed which files were live. Schema drift slipped bad records into production datasets. There was no reliable time travel because the lake had no authoritative version history.

The underlying files were immutable, but the business object was mutable: a table whose rows change, whose schema evolves, whose consumers need isolation, and whose history matters. Delta Lake's move was to stop treating the directory listing as the source of truth.

2 · Task — make Parquet tables transactional without owning storage

The goal was not to replace the lake with a database. The goal was to keep the cheap object-store data plane and add a table control plane strong enough for production analytics.

That meant Delta had to provide a serial history of versions, atomic commits, optimistic concurrency, schema validation, data skipping metadata, and maintenance commands while still storing data as open Parquet files. It also had to work with batch and streaming, because lakehouse workloads are rarely one or the other. The format needed to be self-describing enough that a copied table directory could still be read without a metastore, but structured enough that newer features could declare protocol requirements explicitly.

  1. Durable table state — the log, not the object-store directory listing, defines which files are live.
  2. Transactional writes — writers append one atomic commit after writing data files optimistically.
  3. Interoperable reads — Spark first, then a path for Rust, Python, Java Kernel, Trino, Flink, and catalog-managed engines.

3 · Action — how Delta actually works

Six lenses: the log, concurrency, time travel, DML and schema control, layout maintenance, and the newer interoperability layer.

3.1 The transaction log — JSON commits plus checkpoints

Every Delta table has a _delta_log/ directory. The atomic unit is a zero-padded JSON file such as 00000000000000000045.json. It is newline-delimited JSON, where each line is an action: add a Parquet file, remove a file with a tombstone, change metaData, update protocol, record txn application ids, or attach commit provenance through commitInfo. The data files themselves remain Parquet; the log says which ones are part of version 45.

Reading a table means finding the latest complete snapshot: start from a checkpoint, replay JSON commits after it, then produce the set of live files and metadata. Checkpoints are Parquet files such as 00000000000000000042.checkpoint.parquet, tracked by _last_checkpoint, so large tables do not replay millions of JSON actions on every read. Newer protocol text also describes v2 checkpoints and sidecar files for very large action sets.

This is the core reason Delta works on object storage: the expensive and failure-prone operation is reduced to publishing the next log entry, while data files can be written before the commit and garbage-collected later if abandoned.

3.2 Optimistic concurrency and snapshot isolation

Delta uses MVCC. Writers read a snapshot, write new Parquet files optimistically, then try to commit the next version. If another writer won the version first, Delta checks whether the two operations conflict: appends to disjoint partitions can often proceed after retry; overlapping updates, deletes, or metadata changes can fail. Readers do not block writers and writers do not rewrite in-place data. A reader holds a consistent snapshot even while later commits land.

The caveat is storage semantics. Delta's ACID guarantees assume atomic visibility, mutual exclusion for creating the next log file, and consistent listing. Cloud object stores have improved, but multi-writer and multi-cluster deployments still need careful configuration: DynamoDB-backed S3 log stores, catalog-managed tables, or a commit coordinator for paths where "just write the next JSON file" is not sufficient.

3.3 Time travel — version and timestamp as APIs

Because the log is an ordered history, older table states are addressable. Spark reads can specify versionAsOf or timestampAsOf; streaming can start from startingVersion or startingTimestamp; DESCRIBE HISTORY exposes versions, timestamps, operations, read versions, isolation level, and operation metrics. This is not a backup system by itself. It is bounded by log retention and VACUUM, whose default retention is seven days for unreferenced files.

3.4 Schema enforcement, evolution, MERGE, and DML

Delta enforces schema at write time, which is the unglamorous feature that saves real pipelines. Bad records do not silently widen a table unless schema evolution is explicitly enabled through options such as mergeSchema or auto-merge configuration. The same transaction mechanism powers MERGE INTO, UPDATE, DELETE, generated columns, constraints, and change data files under _change_data/. A MERGE is not editing a Parquet file in place; it is writing replacement files or deletion-vector metadata and committing adds/removes atomically.

Streaming is first-class in the Spark implementation. Delta tables are both sources and sinks for Structured Streaming, with checkpoints providing exactly-once semantics when concurrent batch jobs or multiple streams touch the same table. The tricky part is downstream semantics for non-append commits: options such as skipChangeCommits exist because a stream cannot always propagate arbitrary rewrites as simple appended rows.

3.5 OPTIMIZE, Z-order, liquid clustering, deletion vectors

Lakehouse tables fail operationally when they become small-file museums. Delta's answer is explicit maintenance. OPTIMIZE compacts small Parquet files; ZORDER BY co-locates related values for data skipping; liquid clustering replaces rigid partition/Z-order choices with clustering columns that can evolve over time, then incrementally reclusters with OPTIMIZE. Liquid clustering is not compatible with partitioning or Z-order on the same table, and the docs cap clustering columns at four.

Deletion vectors are the merge-on-read feature: instead of rewriting a large Parquet file because one row was deleted, Delta can record a bitmap-like vector of invalidated rows and apply it during reads. This lowers write amplification for DELETE, UPDATE, and MERGE, but it shifts work to readers and maintenance. Physical purge still requires file rewrite through OPTIMIZE or REORG TABLE ... APPLY (PURGE).

3.6 delta-rs, Delta Kernel, and UniForm interop

The main repo remains Scala/Java with Spark as the reference path, but the project is clearly trying to escape "Spark format" status. delta-rs is a native Rust implementation with Python bindings used by engines such as DataFusion and by lighter-weight data tools. Delta Kernel provides Java APIs — Table, Snapshot, Scan, Transaction, and the pluggable Engine interface — so connector authors can use protocol logic without reimplementing every rule. The separate Rust/C Kernel effort is newer and explicitly experimental, with reads and limited writes still maturing.

UniForm attacks the table-format war from another angle. Delta Universal Format writes Iceberg and Hudi metadata for a Delta table so Iceberg/Hudi readers can access the same Parquet data. Useful, yes. Free, no. It requires table features such as column mapping and writer protocol support; Iceberg/Hudi metadata generation is asynchronous; write support remains Delta-side; and the docs call out limitations such as UniForm not working with deletion vectors enabled.

4 · Result — what you get in practice

The value is a set of database-like behaviors over object storage, plus the operational chores that come with pretending files are rows.

The best mental model: Delta gives the lake a commit log. Everything powerful comes from that. Everything painful comes from keeping that log, the Parquet files, the object store, and multiple engines in agreement.

5 · Takeaways — how to read Delta Lake in 2026

Five calibrated conclusions for teams choosing a lakehouse table format.

What's genuinely strong

  • The core architecture is simple and durable. JSON actions plus Parquet checkpoints is understandable, inspectable, and proven at scale.
  • Spark integration is mature. If Spark Structured Streaming and Spark SQL DML are central to your platform, Delta is deeply paved.
  • Operational commands are real product surface. DESCRIBE HISTORY, VACUUM, OPTIMIZE, clustering, and DML are not side projects.
  • The protocol is explicit. Reader/writer versions and table features make compatibility visible, even when they are annoying.

What to budget for

  • Maintenance jobs. Compact small files, vacuum old data, monitor checkpoints, and keep retention aligned with rollback needs.
  • Protocol discipline. A table feature enabled by one writer can make another engine unable to read or write the table.
  • Interop testing. UniForm, Kernel, and delta-rs reduce lock-in, but they do not erase semantic differences between engines.

Use it when Spark is the write authority

Delta is easiest to justify when Spark owns ingestion, streaming, and DML, and other engines are mostly readers. That is the path where the implementation is most mature.

Be careful with multi-cluster object-store writes

On S3 especially, treat write coordination as architecture, not configuration trivia. Use supported log stores, catalog-managed commits, or a commit coordinator where required.

Do not confuse open files with open semantics

Parquet data is portable; the table semantics live in the log and protocol. A reader that ignores deletion vectors, column mapping, or feature flags is not a Delta reader in any meaningful sense.
