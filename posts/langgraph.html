LangGraph

LangGraph is the LangChain team's low-level answer to a problem every serious agent eventually hits: a useful agent is not a single prompt call, and not always a neat DAG. It is a loop with tools, retries, checkpoints, approvals, streaming, memory, and state that must survive between turns. The good idea is to make that control flow explicit as graphs/state, then run it with durable semantics. The tradeoff is equally real: if your agent is still a dozen lines of Python, LangGraph may be architecture before need.

In short

A graph runtime for agents that need explicit control flow, durable state, and human checkpoints — not just a prettier wrapper around an LLM call.

What it is

An MIT-licensed, low-level orchestration library for building stateful, multi-actor LLM applications. In Python, the center is StateGraph: nodes are ordinary functions or runnables, edges choose what runs next, and a shared typed State carries updates across the graph. There is also a JS/TS port, langgraphjs. It lives in the LangChain ecosystem, but the docs and README are explicit: LangGraph can be used without LangChain.

Why it's interesting

It makes cycles a first-class feature. A ReAct agent is not a pipeline; it is "call model → maybe call tool → call model again → maybe stop." LangGraph models that directly with conditional edges, loops, reducers, checkpoints, interrupts, and streaming. The compiled graph runs on a Pregel-inspired runtime with super-steps, inactive/active nodes, and message passing, which gives the library a real execution model rather than just callback spaghetti.

Biggest concerns

The abstraction has gravity. You inherit graph vocabulary, checkpoint semantics, LangChain-adjacent dependencies, version churn, and a state-debugging model that is harder than reading a plain while loop. The open library is useful, but the polished deployment, Studio visualization, tracing, and hosted persistence story pulls toward LangSmith / LangGraph Platform. That boundary matters when teams adopt it as infrastructure.

Verdict

LangGraph earns its keep when the workflow has durable state, branching, cycles, approval gates, retries, parallel actors, or a need to resume exactly where it stopped. It is a strong fit for production agents that are closer to small distributed systems than scripts. For a simple tool-calling bot, start with a plain loop or a higher-level agent factory; add LangGraph when the state machine is already there in your head and the framework lets you make it visible.

First — a graph you can run

LangGraph is easiest to grasp as a small state machine you assemble in code. Here is the canonical agent loop — a model node that either calls a tool or stops — in full:

from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]    # reducer: append, don't overwrite

def agent(state: State):
    return {"messages": [llm_with_tools.invoke(state["messages"])]}

def route(state: State):                        # control flow is just a function
    last = state["messages"][-1]
    return "tools" if last.tool_calls else END

builder = StateGraph(State)
builder.add_node("agent", agent)
builder.add_node("tools", tool_node)
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route, {"tools": "tools", END: END})
builder.add_edge("tools", "agent")             # the cycle: tool result returns to the agent
graph = builder.compile()

Running it streams each super-step's state update as the model and tools take turns:

for step in graph.stream({"messages": [("user", "What is 17% of last month's revenue?")]}):
    print(step)
# {'agent': {'messages': [AIMessage(tool_calls=[{'name': 'lookup_revenue', ...}])]}}
# {'tools': {'messages': [ToolMessage(content='revenue=842000')]}}
# {'agent': {'messages': [AIMessage(content='17% of $842,000 is $143,140.')]}}

Three things in that snippet are the whole framework: a typed state with a reducer, an edge that is a function of state, and a cycle from tool back to agent that a DAG could not express. Everything below is those three ideas plus a runtime that can persist and resume them.

1 · Situation — agents stopped looking like chains

The original LangChain mental model was a chain: take an input, transform it, call a model, parse the output, maybe call a tool. That shape worked for early demos and deterministic pipelines. It breaks down once the application has to keep operating after the first answer.

Real agents loop. The model proposes a tool call, the tool produces evidence, the model decides whether it has enough, a guardrail may reject the action, a human may approve it, and the run may need to pause for hours before resuming. A DAG can represent some of that with enough nodes, but DAG-only execution fights the shape of the problem. Retrying a failed step, returning from a tool to the same planner, or letting two specialist agents update shared context is easier to express as a cyclic graph than as a linear chain with hidden control flow.

The second pressure is durability. A chat assistant is not a stateless HTTP request once it can spend money, schedule work, modify tickets, or wait for a human. The state of the run has to be inspectable, persistable, replayable, and sometimes editable. That means a framework needs more than prompt templates; it needs a runtime, a state model, and a persistence model.

2 · Task — make the control flow explicit without stealing the user's architecture

LangGraph's task is narrow and ambitious: provide the low-level mechanics for long-running, stateful agents while leaving the nodes themselves as ordinary code.

The library does not require every node to be an LLM. A node can be a function, an agent, a tool wrapper, a validator, a retriever, a human approval gate, or deterministic business logic. Edges decide which node receives control next. State is a typed shared object, usually a TypedDict, dataclass, or Pydantic model. Reducers define how node updates are merged. The user still owns the architecture; LangGraph supplies the graph compiler, runtime, streaming surface, checkpoints, and resumability.

That design target explains both the appeal and the cost. LangGraph is lower-level than CrewAI-style role orchestration and lower-level than LangChain's new create_agent. It is not trying to hide the state machine. It is trying to make the state machine the product.

3 · Action — how LangGraph actually works

The implementation is easier to reason about if you separate the programming model from the runtime contract.

3.1 StateGraph: nodes, edges, conditional edges, cycles

A LangGraph application usually starts with from langgraph.graph import StateGraph, START, END. You instantiate StateGraph(State), register nodes with add_node, connect fixed transitions with add_edge, and route branches with add_conditional_edges. The compiled graph is then invoked or streamed. Unlike Airflow-style DAGs, cycles are not a workaround; the graph can intentionally route from a tool node back to an agent node until a stop condition returns END.

This is the key move: control flow is data. The route function can inspect state and return the next node name. The diagram that Studio renders is not documentation pasted on top of code; it is close to the program's execution structure.

3.2 Typed State + reducers

Every graph has a shared State. In the docs, the canonical examples use TypedDict; Pydantic and dataclasses are also supported. Nodes do not return the whole state. They return partial updates, and LangGraph applies those updates channel-by-channel.

The important detail is reducers. If a key has no reducer, a node update overwrites it. If the key is annotated with a reducer — for example Annotated[list[str], operator.add] or LangGraph's add_messages helper — updates are merged. MessagesState is the prebuilt message-oriented state used in quickstarts, and add_messages handles appending, ID-aware replacement, and message deserialization. This is small but load-bearing: without reducers, parallel branches and iterative message history become accidental last-write-wins bugs.

from typing import Annotated, TypedDict
import operator

class State(TypedDict):
    messages: Annotated[list, add_messages]        # merge: append + ID-aware replace
    visited:  Annotated[list[str], operator.add]   # merge: concatenate
    answer:   str                                  # no reducer: last write wins

# two parallel branches each return {"visited": ["a"]} and {"visited": ["b"]}
# -> the reducer concatenates to ["a", "b"] instead of one clobbering the other

3.3 The Pregel-style runtime / super-steps

After you call compile(), the graph is checked and converted into an executable object. The docs describe the runtime as inspired by Google's Pregel: nodes communicate by message passing, execution advances in discrete super-steps, active nodes run when they receive input on a channel, and nodes vote to halt when they have no incoming messages. The run terminates when all nodes are inactive and no messages are in flight.

That model matters for parallelism. Two branches from START can run in the same super-step and write updates that reducers reconcile. It also matters for debugging: a LangGraph trace is not just "function A called function B." It is a sequence of state snapshots, node activations, emitted updates, routed edges, and checkpoint writes.

3.4 Persistence/checkpointers & durable execution

Persistence is the feature that makes LangGraph feel less like a graph helper and more like agent infrastructure. A checkpointer persists a thread's graph state as checkpoints; a store persists application-defined data across threads. The docs separate the two: checkpointers are short-term, thread-scoped memory for conversation continuity, fault tolerance, time travel, and human-in-the-loop; stores are long-term, cross-thread memory for user facts or shared knowledge.

The API shape is direct: compile with builder.compile(checkpointer=checkpointer, store=store), then pass {"configurable": {"thread_id": "thread-1"}} at invocation time. That thread_id becomes the persistent cursor. If the process dies, the next run can reload the thread's checkpoint rather than restart the agent from the beginning.

from langgraph.checkpoint.memory import MemorySaver
graph = builder.compile(checkpointer=MemorySaver())

cfg = {"configurable": {"thread_id": "ticket-4821"}}
graph.invoke({"messages": [("user", "open a refund")]}, cfg)
# ... process restarts ...
graph.invoke({"messages": [("user", "actually, make it partial")]}, cfg)
# same thread_id -> the second call resumes the saved conversation, not a fresh one

3.5 Human-in-the-loop interrupts & time travel

LangGraph's interrupt() is the cleanest example of why durable execution matters. A node can call interrupt(payload), the runtime saves state through the checkpointer, surfaces the payload to the caller, and waits indefinitely. Later, the caller resumes with Command(resume=...) using the same thread_id; that value becomes the return value of interrupt() inside the node.

That gives approval workflows, review-and-edit steps, and tool-call gates a first-class representation. It also creates time-travel workflows: inspect checkpoints, fork from an earlier state, edit state, or replay a different branch. The caveat is subtle but real: when an interrupt resumes, code before the interrupt in that node can run again, so nodes need idempotence discipline around external side effects.

from langgraph.types import interrupt, Command

def approve(state: State):
    decision = interrupt({"action": "refund", "amount": state["amount"]})  # pauses here
    return {"approved": decision == "yes"}

# caller side: run until the interrupt, then resume — possibly hours later, new process
graph.invoke(inputs, cfg)                  # stops at interrupt, state checkpointed
graph.invoke(Command(resume="yes"), cfg)   # interrupt() returns "yes"; the node continues

3.6 Streaming, subgraphs, prebuilt react agent

LangGraph exposes both classic stream / astream modes and newer event streaming. Current docs recommend graph.stream_events(..., version="v3"), which gives typed projections such as messages, values, subgraphs, interrupts, and output. The older stream-mode API still exposes updates, values, messages, custom, checkpoints, tasks, and debug. In other words, you can stream tokens, state updates, checkpoint events, and debugging traces without inventing your own event bus.

Subgraphs let a node contain another graph, which is how larger systems can decompose specialist workflows without flattening everything into one unreadable canvas. The package also includes prebuilt helpers. Historically the headline one was langgraph.prebuilt.create_react_agent, a compiled graph that loops model/tool calls until completion; in v1 docs and source it is deprecated in favor of langchain.agents.create_agent, but the implementation is still useful as a reference for how LangGraph expresses the canonical agent loop.

4 · Result — what this buys in practice

LangGraph is strongest when its mechanisms line up with real operational pain.

The result is not magic intelligence. LangGraph does not make the model reason better. It makes the control plane around a model explicit enough that a team can inspect, resume, branch, and operate it.

5 · Takeaways — where I would use it

Five calibrated conclusions for engineers deciding whether the graph should exist.

Use it when state is the product

  • Long-running agents. If a run may pause, fail, resume, or wait on a human, the checkpointer model is the point.
  • Non-linear control flow. Tool loops, validators, retries, specialist agents, and fallback branches map cleanly to conditional edges and cycles.
  • Team debugging. State snapshots and graph visualization are valuable when multiple engineers need to reason about why an agent took a path.

Be cautious when the agent is still small

  • A plain loop is a serious baseline. A while loop with clear variables, logs, and tests is often more legible than a graph with three nodes.
  • LangChain gravity is real. You can use LangGraph standalone, but examples, integrations, observability, and deployment naturally pull toward the larger stack.
  • APIs move. The v1 era is cleaner, but create_react_agent deprecation and streaming API evolution are reminders to pin versions and read migration notes.

Design reducers before adding nodes

Most LangGraph bugs I would expect in production are not "the edge points to the wrong node." They are state bugs: a message list overwrote instead of appended, a parallel branch lost a value, a private channel leaked into streaming, or an interrupt resumed after a non-idempotent side effect.

Keep commercial boundaries explicit

The open-source library is MIT and independently useful. The most comfortable production experience likely includes LangSmith / LangGraph Platform features: Studio, hosted deployment, tracing, persistence plumbing, and visual debugging. That may be fine. It should be a conscious platform decision, not an accidental import path.

Think of it as a state-machine runtime

The right comparison is not "does this call OpenAI faster." It is "do I need a durable state machine around an unreliable stochastic worker." If yes, LangGraph is one of the most mature open-source answers.
