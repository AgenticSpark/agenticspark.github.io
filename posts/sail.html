In short

Four pillars in the tagline, four very different maturity levels — defended below with code and the project's own docs.

What it is

A JVM-free Rust reimplementation of Spark's engine on DataFusion + Arrow. It implements the Spark Connect protocol, so SparkSession.builder.remote("sc://host:50051") is the only change to run PySpark on it. Local single-process mode + a distributed cluster mode (actor control plane, Arrow Flight data plane). Apache-2.0, backed by the company LakeSail.

Why it's interesting

The most architecturally correct "post-JVM Spark" bet of 2026: 36 clean crates, a custom in-house Spark-SQL parser, zero-copy PyO3 Python UDFs, native Delta + Iceberg, five catalogs, seven storage backends. The efficiency story is real — a 100GB shuffle with zero spill where Spark spilled 110GB, plus near-instant worker startup.

Biggest concerns

Three of four tagline pillars run ahead of the code. Streaming is bounded/windowless (not_impl_err!("streaming window")) and the docs contradict themselves on its status. "Multimodal AI engine" is binaryFile reads + Python UDFs + an MCP server. "94% cheaper" is an inferred instance-downsizing claim from a single-node TPC-H. Pre-1.0 (v0.6.3).

Verdict

The most credible and most architecturally correct open-source "post-JVM Spark" of 2026 — DataFusion + Arrow + Spark Connect is exactly the right thesis, and the engineering behind it is real and large. Worth piloting today for batch SQL/DataFrame ETL where you own the code and can diff the output against Spark; the lakehouse/catalog breadth makes it usable against a real data estate now. But treat "4× / 94% cheaper" as a best-case single-node ceiling, and treat "streaming + AI + multimodal" as roadmap, not shipped pillars. Solid pre-1.0 (v0.6.3, ~2.5 years old, 2.8k stars, Apache-2.0, company-backed).

1 · Situation — Spark's JVM tax, and a Rust data stack that grew up

By 2026, Spark is both the planet's default distributed engine and the thing everyone is trying to escape.

The complaints about Apache Spark are old and well-rehearsed: the JVM brings garbage-collection pauses that spike tail latency, multi-second executor startup that makes elastic and serverless workloads painful, and a memory model that spills to disk under pressure. A whole category of "Spark accelerators" exists to paper over this — Databricks' proprietary Photon, the open-source DataFusion Comet and Gluten + Velox — each swapping Spark's row-oriented JVM operators for vectorized native ones while keeping the JVM driver and the Spark codebase underneath.

Sail makes a more radical bet: don't accelerate Spark, replace it. Throw away the JVM entirely, reimplement the engine in Rust on Apache DataFusion (a Rust-native query engine) + Arrow, and keep compatibility not at the bytecode level but at the protocol level — Spark Connect, the gRPC API Spark 3.4+ exposes so clients can talk to a remote driver. If you speak Spark Connect fluently, a PySpark program cannot tell whether the other end of the socket is a JVM or a Rust binary. That is the wedge Sail drives in.

The timing isn't an accident. DataFusion has matured into a production-grade, extensible SQL engine with a vectorized Arrow core; Arrow is now the lingua franca of columnar data; and Rust's data ecosystem (object_store, the Arrow crates, tokio) is finally rich enough to build a real distributed engine on. Sail is what you get when a well-funded team decides the substrate is finally ready.

Apache Spark — the JVM baseline

Mature, ubiquitous, full RDD + Structured Streaming. Sail targets its JVM tax: GC pauses, slow startup, disk spill.

Comet / Gluten — accelerate under the JVM

Native vectorized operators beneath Spark's JVM driver. Lower-risk, less radical; the JVM stays.

Photon — proprietary accelerator

Fast but closed and Databricks-locked. Sail is open and engine-replacing, not Spark-accelerating.

Sail — replace the engine entirely

lakehq/sail (2.8K ⭐, Apache-2.0, Rust) — a JVM-free DataFusion engine behind the Spark Connect protocol. The subject of this report.

2 · Task — four promises, not all at the same maturity

The tagline makes four commitments. Separating them up front is the whole point of this report.

Sail's one-sentence pitch, verbatim from the README.md: "a drop-in Apache Spark replacement written in Rust, unifying batch processing, stream processing, and compute-intensive AI workloads on a distributed, multimodal compute engine." That sentence makes four promises:

  1. Drop-in Spark compatibility — PySpark SQL + DataFrame code runs unmodified via Spark Connect, "no code rewrites required."
  2. Batch processing — a faster, cheaper batch engine than Spark.
  3. Stream processing — unified streaming alongside batch.
  4. Compute-intensive AI / multimodal — a "multimodal compute engine" for AI workloads.

The README backs this with hard numbers: "~4x faster (up to 8x in specific workloads) than Spark and 94% cheaper on infrastructure costs" (README L12), and "Proven on ClickBench, outperforming Spark, popular Spark accelerators, Databricks, and Snowflake."

The thesis of this report, stated plainly: promises (1) and (2) are real and well-built; promise (3) is genuinely under construction but oversold by the tagline; promise (4) is positioning around real-but-modest plumbing. A reader deciding whether to bet on Sail needs to know which pillars bear weight today — checked against the source at commit d4971209.

3 · Action — what's actually in the repository

Eight lenses: the crate graph, Spark Connect, cluster mode, the batch benchmark, streaming, AI/multimodal, lakehouse, project health.

3.1 The shape — 36 crates, ~257K lines of Rust, clean separation

Sail is not a thin wrapper. The workspace is 36 crates (crates/) and roughly 257,000 lines of Rust, decomposed with the kind of module discipline that signals a team intending to maintain this for years: a custom SQL front-end (sail-sql-parser, sail-sql-macro, sail-sql-analyzer — built from parser combinators + proc macros, not a fork of someone else's grammar); a full planning stack (logical / physical plan + optimizers); execution + distribution (sail-execution, sail-flight); and the protocol surface (sail-spark-connect).

The engine rests on Apache DataFusion 53.1 (Cargo.toml L156) and the Arrow Rust stack. Sail does not reimplement vectorized execution from scratch — it composes DataFusion's operators and adds the Spark semantics, the distribution layer, the lakehouse integrations, and the protocol surface on top. That is the right build-vs-buy call, and it is why chasing Spark parity is plausible at all.

3.2 Compatibility — Spark Connect is the whole trick

Sail implements the Spark Connect gRPC service: a PySpark client builds an unresolved logical plan as protobuf, ships it to the server, and Sail translates that into its own logical plan, optimizes, and executes. The entire client-side change (migration guide):

# before
spark = SparkSession.builder.getOrCreate()
# after — the only change
spark = SparkSession.builder.remote("sc://localhost:50051").getOrCreate()

That is the genuinely elegant part. Compatibility is defined by a protocol, not by reproducing Spark's internals — so Sail is free to be a completely different engine underneath while remaining a faithful endpoint. Ibis' PySpark backend works through the same one-line change.

3.3 Cluster mode — actor-model control plane, Arrow Flight data plane

Local mode is a single process (the common case, and what the benchmarks use). Cluster mode (architecture doc) is a real distributed system: a server hosts multiple sessions, each driven by a cluster job runner that owns a driver scheduling a physical plan as stages and tasks across workers. This is a textbook-correct design for a 2026 distributed query engine — and because workers are JVM-free Rust processes, a worker is a static binary that boots in milliseconds, which is exactly what enables the "instant startup / elastic" pitch (arch L38, L40).

3.4 The batch benchmark — strong engine, soft headline

The "~4x faster / 94% cheaper" claim comes from benchmark-results. Methodology, stated honestly in the doc itself: a derived TPC-H (L8) at SF100 (~100 GB), 22 queries, on a single AWS r8g.4xlarge (16 vCPU / 128 GB). Result: total query time 387s → 103s ("nearly 4x faster"), and shuffle spill 110 GB → 0 GB.

Two things are simultaneously true. The engine is legitimately fast and efficient — a 3.8× wall-clock win and zero shuffle spill where Spark spilled 110 GB is a real result reflecting DataFusion's vectorized core and Rust's memory model. But the headline oversells in three specific ways:

  • "4×" is an average across a very wide range. Per-query speedups span roughly 43%–727% — a handful of queries dominate the mean. "Up to 8x in specific workloads" is true; so is "barely faster on some."
  • "94% cheaper" is an inference, not a measurement. The doc's own logic (L28): "Sail can run on 1/4 the instance size, leading to up to 94% cost reduction." A derived instance-downsizing claim, not a measured cloud bill.
  • It's single-node and single-vendor. A 100 GB TPC-H on one box is a good engine benchmark but a weak proxy for the distributed, multi-tenant reality where Spark actually hurts — and the baseline Spark spilled 110 GB "despite abundant available memory," hinting at an under-tuned comparator. ClickBench "outperforming Databricks, Snowflake" links to a vendor-hosted redirect.

None of this means the numbers are wrong. It means the honest headline is: "on a single-node 100 GB TPC-H, Sail ran ~3.8× faster with zero shuffle spill, which on paper lets you use a smaller instance." Still excellent. Just not the same sentence as "4x faster and 94% cheaper" full stop.

3.5 Streaming — the most oversold pillar (and the most interesting nuance)

Here the tagline and the code disagree, and the docs don't even agree with each other. The README sells "stream processing" as present-tense. But the marketing why-sail page (L16) says latency spikes "will also be eliminated when Sail supports data streaming in the near future," while the migration guide (L67) marks Structured Streaming "✅ (partial)." Three docs, three different maturity signals.

The code resolves it: more real than vaporware, less than shipped. There is a substantial streaming subsystem — 21 Rust files across six crates: an event model in sail-common-datafusion, streaming node variants in logical + physical plan, and a StreamingRewriter in rewriter.rs. But that rewriter is candid about its gaps:

struct StreamingRewriter;                              // L31
// TODO: support BarrierNode for streaming properly.   // L52
not_impl_err!("streaming window: {plan:?}")          // L86

So streaming today is bounded / early: real source adapters and a real rewrite path, but windowing is not implemented and barrier handling is a TODO. Structured Streaming's whole point — windowed aggregations, watermarks, stateful operators over unbounded input — is exactly the part that isn't there yet. The infrastructure is genuine and the direction clear, but a reader who expects to port a windowed Spark Streaming job today will be disappointed. The honest label: "streaming engine under active construction; bounded queries work, windowing pending."

3.6 "AI / multimodal" — good plumbing, aspirational framing

The fourth pillar is where marketing most exceeds substance, though what is there is real and useful. "Multimodal / AI" decomposes into three concrete things:

  • Reading binary/multimodal data. The getting-started doc (L85) shows the whole story: spark.read.format("binaryFile").option("pathGlobFilter","*.png") and a SQL table USING binaryFile over s3://…/*.pdf. Genuinely useful for AI pipelines — but it's ingesting image/PDF blobs, not multimodal compute.
  • Python UDFs via PyO3. sail-python-udf is the real strength and it's comprehensive — UDFs, UDAFs, UDTFs, group-map and map-iterator UDFs (udf/) bridged into the Arrow engine via PyO3. Run arbitrary Python (incl. ML inference) close to the data with Arrow-native exchange. The legitimately good AI-adjacent feature.
  • An MCP server. sail spark mcp-server (docs) exposes Spark SQL as Model Context Protocol tools so an LLM agent can query datasets in natural language. A nice bridge — not a compute capability.

What is not in the code: any GPU/tensor/embedding/torch engine (a grep across the crates finds none). So "multimodal compute engine" is positioning. The truthful version: "Sail can ingest binary/multimodal files and run your Python/ML code on them via zero-copy Arrow UDFs, and expose its SQL to LLM agents over MCP." A respectable AI-data-plumbing story — not an AI compute engine, which is what the tagline implies.

3.7 Lakehouse and storage — quietly the most complete part

Ironically the pillar the tagline under-sells. Native Delta Lake (sail-delta-lake) and Iceberg (sail-iceberg); a catalog matrix covering AWS Glue, Databricks Unity, Hive Metastore, Iceberg REST, and Microsoft OneLake (catalog crates); and an sail-object-store layer reaching S3, Azure, GCS, HDFS, Hugging Face, Cloudflare R2 and HTTP. For an org whose real question is "can this read my existing lakehouse tables from my existing catalog," the answer is unusually often "yes" — and that, more than streaming or AI, is what makes Sail a plausible Spark substitute for batch ETL today.

3.8 Project health — company-backed, concentrated but not fragile

Sail is backed by LakeSail (lakesail.com), which changes the risk calculus versus a solo project. Contribution is concentrated — the top contributor is ~34% of ~1,300 commits, top three/four carry the bulk — but it's a team, with 23 releases and a steady cadence (v0.6.3, 2026-05-21). Numbers: 2,830 stars, 163 forks, 226 open issues, Apache-2.0, ~2.5 years old. The honest maturity read is simply pre-1.0: the API surface and especially cluster mode are younger than local mode; treat v0.6 as "serious pilot, not yet bet-the-farm."

The resource-efficiency wins are the part most likely to matter in production: a JVM-free worker boots near-instantly (good for elastic/serverless), and an engine that finishes a 100 GB shuffle without spilling to disk where Spark spilled 110 GB both speeds up jobs and shrinks the instance you need. If your pain with Spark is GC tail latency, slow startup, and memory pressure on batch jobs, Sail targets exactly those pains and appears to land hits.

The caveats are equally concrete, and the migration guide names them plainly: no RDD (JVM-only, outside Spark Connect, L61), Pandas-on-Spark is roadmap, many Spark configs silently no-op, error messages differ, and storage config is entirely different (your s3a Hadoop settings don't apply). The compatibility checker is explicitly experimental and only verifies whether functions are implemented, not behavioral parity, and doesn't cover SQL strings (L35). The real adoption path is "point a copy of your batch job at Sail, run it on a representative dataset, and diff the output" — which the guide, to its credit, recommends.

5 · Takeaways — how to read Sail in 2026

Five calibrated conclusions for an engineer evaluating a Spark replacement.

What's genuinely strong

  • The right architecture. DataFusion + Arrow + Spark Connect is the correct post-JVM Spark thesis, executed with real engineering — 36 crates, a custom SQL parser, an actor/Flight cluster design. Not a toy.
  • Batch efficiency is believable. ~3.8× and zero-spill on a single-node TPC-H is a great result that reflects the engine, not the marketing.
  • Lakehouse breadth is the sleeper strength. Native Delta + Iceberg and five catalogs make Sail usable against an existing data estate now.
  • Zero-copy Python UDFs (PyO3) are comprehensive and the real AI-adjacent win.

What to discount

  • The "4× / 94%" headline. Averaged, partly-inferred, single-node, single-vendor. Benchmark your workload.
  • Streaming as a shipped pillar. Real infra but windowing-incomplete, and the docs contradict themselves on status.
  • "Multimodal AI compute engine." It's binaryFile reads + Python UDFs + MCP — good plumbing, not a compute engine.
  • "Drop-in" as turnkey. No RDD, partial streaming, silent config no-ops, experimental compat checker — validate before betting production.

Pilot it where you control the code

Best first bet: a batch SQL/DataFrame ETL job, run side by side with Spark, output diffed. Worst first bet: a windowed streaming pipeline or an RDD-dependent job.

Lead with the lakehouse question

Before chasing the benchmark, confirm Sail reads your tables from your catalog (Glue / Unity / Hive / Iceberg REST / OneLake). That's where adoption succeeds or stalls.

Treat v0.6 as a pilot band

Company-backed and moving fast, but pre-1.0 — cluster mode is younger than local mode. Serious to trial, not yet to bet the farm on.
