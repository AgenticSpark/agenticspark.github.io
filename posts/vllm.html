LLM inference engines · STAR analysis

vLLM

The open-source inference engine that turned a memory-management idea — PagedAttention — into the default way the world serves large language models. One paper from UC Berkeley in 2023 became 83K stars, an OpenAI-compatible server every framework targets, and a near-complete V1 rewrite of the engine core. The throughput story is real and the hardware reach is enormous, but the codebase moves fast enough to draw blood: 2,000 open issues, a long tail of model × quantization × backend combinations, and memory tuning that still surprises newcomers. Here is what's actually inside at v0.23, and where the speed comes from.

In short

One idea — borrow virtual memory from operating systems and apply it to the KV cache — and a community that turned it into infrastructure.

What it is

A high-throughput LLM inference and serving engine. It runs open models (Llama, Qwen, Mistral, DeepSeek, Gemma, multimodal VLMs, MoEs) behind an OpenAI-compatible HTTP server (vllm serve) or an offline LLM() Python class. The headline mechanism is PagedAttention: the KV cache is stored in fixed-size, non-contiguous blocks mapped through a block table, the way an OS pages physical memory. Apache-2.0, born at UC Berkeley's Sky Computing Lab, now hosted under the PyTorch Foundation.

Why it's interesting

It solved the bottleneck that capped LLM serving throughput — KV-cache memory fragmentation — and paired it with continuous (iteration-level) batching so the GPU never idles waiting for the slowest sequence in a batch. The SOSP'23 paper reported up to 24× the throughput of HuggingFace Transformers. On top of that: automatic prefix caching, chunked prefill, speculative decoding, AWQ/GPTQ/FP8 quantization, tensor + pipeline + expert parallelism, and a from-scratch V1 engine with an isolated core process and a near-zero-overhead scheduler.

Biggest concerns

Velocity is the cost. ~2,000 open issues, frequent breaking changes, and a combinatorial support matrix (each model × quant scheme × attention backend × accelerator is its own correctness surface). gpu_memory_utilization and the V0→V1 migration still trip people. The project is enormous and load-bearing for a whole industry, which makes "it broke on my exact config" both common and high-stakes.

Verdict

The default answer to "how do I serve an open LLM with good throughput," and deservedly so — PagedAttention plus continuous batching is one of the highest-leverage systems ideas of the LLM era, and the V1 rewrite shows the team is willing to pay down its own architectural debt. Adopt it for almost any open-weights serving job; the OpenAI-compatible API means you can drop it behind existing clients. But pin your version, benchmark your model and quant on your GPU, and budget for the operational reality of a project that ships fast and supports an almost unreasonable hardware matrix. Treat the headline throughput multipliers as workload-dependent ceilings, not guarantees.

First — one request, end to end

Theory lands better once you have watched the thing run, so here is the whole engine in one launch and one request. vLLM has two front doors; this is the server:

$ vllm serve meta-llama/Llama-3.1-8B-Instruct \
      --max-model-len 8192 \
      --gpu-memory-utilization 0.90 \
      --enable-prefix-caching
INFO  Automatic prefix caching is enabled.
INFO  Using FlashAttention backend.
INFO  GPU KV cache: 42,160 tokens  (2,635 blocks x 16)
INFO  Uvicorn running on http://0.0.0.0:8000

That KV-cache line is the entire design in miniature: after loading the weights, vLLM measured the free HBM, carved it into 2,635 fixed blocks of 16 tokens, and now reports it can hold ~42K tokens of cache across all in-flight requests at once. Send it an ordinary OpenAI request:

$ curl localhost:8000/v1/chat/completions -H 'Content-Type: application/json' -d '{
    "model": "meta-llama/Llama-3.1-8B-Instruct",
    "messages": [{"role": "user", "content": "Name three uses for paged memory."}],
    "max_tokens": 96
  }'

Six things happen between that request and the streamed reply, and the rest of this piece is those six things in detail: the front-end tokenizes the prompt; the scheduler admits it into the current step's batch; the KV-cache manager hands it blocks (reusing any whose contents it has already seen); the executor runs one forward pass over every active sequence at once; the sampler picks the next token; and the loop repeats until an end token or max_tokens, at which point the sequence's blocks are freed for whoever is waiting. The offline path is the same engine minus the HTTP layer:

from vllm import LLM, SamplingParams

llm = LLM(model="meta-llama/Llama-3.1-8B-Instruct", enable_prefix_caching=True)
prompts = ["Name three uses for paged memory."] * 1000      # 1,000 prompts
outs = llm.generate(prompts, SamplingParams(max_tokens=96))  # batched for you
print(outs[0].outputs[0].text)

A thousand prompts, one generate call, and no batching loop in sight — because continuous batching is the engine's native unit of work, not something you orchestrate. Hold these two snippets in mind; every mechanism below exists only to make them fast.

1 · Situation — the KV cache was eating the GPU

By 2023, the thing limiting how many requests a GPU could serve was not compute. It was memory — and specifically how badly the KV cache wasted it.

Every token an autoregressive model generates attends to the keys and values of all previous tokens. To avoid recomputing them, servers cache those K/V tensors — the "KV cache." For a 13B model serving long contexts, that cache can be gigabytes per request, and it grows and shrinks unpredictably as sequences start, generate, and finish. The naive approach reserves one contiguous slab per request, sized to the maximum possible length.

That reservation is catastrophic. A request that might generate 2,048 tokens but stops at 50 holds memory for 2,048. Slabs of different sizes leave unusable gaps between them (external fragmentation), and the unused tail of each slab is dead weight (internal fragmentation). The vLLM paper measured real systems wasting 60–80% of KV memory this way. Wasted memory means a smaller batch; a smaller batch means the expensive GPU sits underutilized; underutilization means low throughput and high cost per token.

The second problem was scheduling. Classic "static" batching groups N requests, runs them together, and can't return the GPU until the longest sequence in the batch finishes — so short requests wait behind long ones and slots go idle as sequences complete at different times.

2 · Task — serve more requests per GPU without rewriting the models

The goal was raw serving efficiency, achieved underneath an unchanged model and a familiar API.

vLLM's bet was that you could fix both problems in the serving layer — memory management and scheduling — without touching model weights or asking users to learn a new programming model. Reuse the same transformer definitions, expose the same OpenAI endpoints, and make the engine underneath dramatically more efficient. Two mechanisms carry the weight, and almost everything else in the project is built around protecting their efficiency across more models, more hardware, and more features.

  1. PagedAttention — eliminate KV-cache fragmentation by paging it like OS virtual memory.
  2. Continuous batching — schedule at every decoding step so finished sequences leave and waiting ones join immediately.

3 · Action — what's actually in the engine

Seven lenses: paged memory, continuous batching, prefix caching, the V1 rewrite, distributed execution, quantization, and the kernel/hardware layer.

3.1 PagedAttention — the KV cache as virtual memory

The KV cache is partitioned into fixed-size blocks (e.g. 16 tokens each). A sequence's tokens map to a list of blocks through a per-sequence block table — exactly an OS page table. Blocks need not be contiguous in GPU memory, so there is no external fragmentation; the only waste is the partially-filled final block of each sequence, which caps internal waste at under one block per request (the paper reports <4%). The attention CUDA kernel (in csrc/) is written to gather K/V from these scattered blocks during the attention computation.

Two consequences fall out for free. First, far less waste means far bigger batches on the same card. Second — and this is the elegant part — blocks can be shared. Two requests with the same prompt prefix, or the parallel samples of one beam search, can point their block tables at the same physical blocks, with copy-on-write when they diverge.

The arithmetic is deliberately boring, which is the point. With a 16-token block, a 41-token prompt takes ceil(41 / 16) = 3 blocks: two full, and one holding 9 live tokens beside 7 slots of bounded slack. The block table is just a list of physical block numbers, so appending a token either writes into the current block's next free slot or, when that block is full, asks the manager for one more block and pushes it onto the list. Nothing is ever copied, resized, or pre-reserved for a length the request may never reach.

3.2 Continuous batching — iteration-level scheduling

Instead of batching whole requests, vLLM batches a single decoding step across whatever sequences are currently active. After each step, finished sequences release their blocks and waiting sequences are admitted, subject to the memory the block manager can hand out. 3.3 Automatic prefix caching — the system-prompt dividend

Because blocks are content-addressable, vLLM hashes block contents and reuses KV blocks across requests that share a prefix. In practice almost every production deployment has a long, identical system prompt (tools, instructions, few-shot examples) in front of every request. With automatic prefix caching enabled, that prefix is computed once and its blocks are reused for every subsequent request — turning a fixed per-request prefill cost into a one-time one. For agent and RAG workloads with heavy shared context, this is frequently a larger real-world win than the raw batching numbers suggest.

Make it concrete. Suppose every request carries the same 1,800-token system prompt — tools, instructions, a few worked examples — which is ~113 blocks of KV. The first request pays to prefill it; every request after that is a cache hit on those blocks and prefills only its own short tail:

POST /v1/chat/completions      # request 1  (cold)
  prompt = SYSTEM(1,800 tok) + "Summarize ticket #4821"
  -> prefill 1,847 tokens

POST /v1/chat/completions      # request 2  (warm)
  prompt = SYSTEM(1,800 tok) + "Summarize ticket #4822"
  -> prefix-cache HIT on 113 blocks; prefill 47 tokens   (97% skipped)

3.4 Chunked prefill and speculative decoding — balancing the two phases

LLM inference has two very different phases: prefill (compute-bound, processes the whole prompt at once) and decode (memory-bound, one token at a time). Mixing them naively means a long prefill stalls everyone's decode. Chunked prefill splits a long prompt into chunks and interleaves them with ongoing decode steps, smoothing latency. Speculative decoding attacks the memory-bound decode phase from the other side: a cheap drafter (a small model, n-gram lookup, or EAGLE/Medusa heads) proposes several tokens, and the big model verifies them in one pass — multiple tokens per forward step when the draft is good.

# draft the next 5 tokens with a small model; the 70B verifies them in one pass
vllm serve meta-llama/Llama-3.1-70B-Instruct \
    --speculative-config '{"model": "meta-llama/Llama-3.1-8B-Instruct",
                           "num_speculative_tokens": 5}'

On a good draft the big model confirms several tokens per forward pass instead of one, so decode-bound, low-concurrency workloads (a single chat stream, say) speed up without changing the output distribution — verification guarantees exactly the tokens the target would have produced on its own. EAGLE and Medusa heads are cheaper drafters that live inside the target model itself; n-gram lookup is the zero-cost option for highly repetitive text.

3.5 The V1 engine — paying down the architecture

vLLM's original engine accreted features until the scheduler and the Python execution path became a measurable overhead at high request rates. The V1 re-architecture (under vllm/v1/, now the default) isolates the performance-critical EngineCore in its own process, separates it from the API/tokenization front-end, and drives a tighter, near-zero-overhead scheduling loop. It also cleaned up multimodal input handling and made prefix caching and chunked prefill first-class rather than bolt-ons. The lesson worth noting: a project this load-bearing chose to rewrite its core rather than keep patching it — uncommon discipline for open-source infrastructure under this much demand.

3.6 Distributed execution — TP, PP, and expert parallelism

For models too big for one GPU, vLLM implements Megatron-style tensor parallelism (shard each layer across GPUs), pipeline parallelism (split layers across GPUs/nodes), and expert parallelism for Mixture-of-Experts models, coordinated through Ray or native multiprocessing. The same --tensor-parallel-size / --pipeline-parallel-size flags scale a deployment from one card to a multi-node cluster without changing the model code.

3.7 Quantization and the kernel layer — the breadth that creates the matrix

vLLM supports a wide spread of quantization formats — AWQ, GPTQ, FP8, INT8, GGUF, bitsandbytes — and a stack of attention backends (FlashAttention, FlashInfer, and hardware-specific paths). The custom kernels live in csrc/, and the engine runs not only on NVIDIA but on AMD ROCm, Intel GPUs/CPUs, AWS Neuron, Google TPU, and CPU. This reach is vLLM's superpower and its tax simultaneously: every (model, quant, backend, accelerator) tuple is a distinct code path that can have its own bug, which is the structural reason the issue tracker is so large. The recent appearance of a rust/ tree and build_rust.sh hints at moving some hot, latency-sensitive plumbing out of Python.

The thing to internalize: vLLM's wins are throughput wins. If your bottleneck is per-request latency at low concurrency, a single inference's wall-clock improvement is real but smaller than the batch-throughput multipliers imply. If your bottleneck is "how many concurrent users can this GPU serve," that's exactly the axis vLLM was built to bend.

5 · Takeaways — how to read vLLM in 2026

Five calibrated conclusions for an engineer choosing a serving stack.

What's genuinely strong

  • The core idea is foundational. PagedAttention + continuous batching is now table-stakes; vLLM is the reference implementation, and the whole field copied it.
  • The OpenAI-compatible server is the integration win. Point any existing client at it and you're done — that compatibility is half of why it won.
  • Hardware reach is unmatched in open source. NVIDIA, AMD, Intel, TPU, Neuron, CPU — few projects span that.
  • The V1 rewrite signals real engineering maturity, not just feature accretion.

What to budget for

  • Version churn. Pin a version in production; read release notes before bumping. Fast-moving means occasionally sharp.
  • The support matrix. "Works on my model/quant/GPU" is not transitive — benchmark and validate your exact tuple.
  • Memory tuning. gpu_memory_utilization, max sequence length, and KV block budget interact; OOMs at load are usually config, not bugs.

Default to it for serving open weights

For throughput-oriented serving of open models behind an OpenAI API, vLLM is the safe default. The burden of proof is on alternatives to beat it on your specific workload.

Match the tool to the bottleneck

Throughput-bound, many users, shared prompts → vLLM shines. Single-stream lowest-latency on one fixed model/GPU → also weigh a compiled engine (TensorRT-LLM) and measure.

Treat the multipliers as ceilings

"24× / Nx" are real headline numbers under specific conditions. Your mileage is set by context length, concurrency, model size, and hardware — measure it.
