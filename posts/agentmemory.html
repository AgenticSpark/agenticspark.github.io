agentmemory

A TypeScript daemon that turns any MCP-speaking coding agent — Claude Code, Codex, Cursor, OpenCode — into one with persistent memory. What it is

A long-running daemon that listens on :3111 and gives MCP-speaking agents persistent memory via 124 REST endpoints, 53 MCP tools, and 12+22 lifecycle hooks for Claude Code and OpenCode. Hybrid retrieval. Apache-2.0.

Biggest strengths

An unusually thoughtful retrieval stack (RRF over BM25 + Vector + Graph with dynamic weight renormalisation), reproducible LongMemEval-S numbers, broad hook coverage, 1,067 passing tests, and senior-grade boot-time invariants — including a vector-dimension mismatch guard that refuses to start rather than corrupt search silently.

Biggest concerns

No committed lockfile despite being a published global CLI (#540). Hidden coupling to a separate iii-engine process from the author's company. Default zero-auth on :3111. Marketing comparisons that mix incompatible benchmarks. Single-maintainer bus factor (320 of 362 commits).

1 · Situation — the agent memory landscape

Persistent memory for LLM agents has become its own software category. Where does this repo actually sit?

Memory for LLM-powered agents has split, by 2026, into two well-defined camps:

Memory as a library / SaaS

Mem0 (56K ⭐, Python, Apache-2.0) and Zep (4.6K ⭐) wrap a vector DB and expose add() / search(). Zep's newer knowledge-graph product, Graphiti, lives in the same lane at 26K ⭐.

Memory inside a full agent runtime

Letta / MemGPT (22.8K ⭐) is the canonical example, with its core-memory / archival split. LangChain and LlamaIndex offer thinner "memory" primitives bolted onto wider frameworks; neither is positioned as a memory system in its own right.

agentmemory stakes out a third position: a memory daemon that runs as a local process, exposes state over REST + MCP + an iii-engine WebSocket, and integrates via hooks into the lifecycle events of coding assistants. The unit of integration isn't from agentmemory import Memory — it's "drop these .mjs hook scripts into your agent config and the daemon will be called on every PostToolUse, SessionStart, Stop event."

That positioning is unusual enough to be worth the deep dive. Whether it is better than Mem0 / Letta / Zep depends entirely on whether you live inside the coding-agent ecosystem it targets.

2 · Task — what it claims to solve

The README's pitch is unsubtle: "Your coding agent remembers everything. No more re-explaining."

Concretely, the headline claims are:

  • 95.2% R@5 retrieval on LongMemEval-S, 88.2% MRR.
  • ~92% token reduction vs pasting full context; ~$10/yr running cost with local embeddings.
  • Zero external databases — "SQLite + iii-engine".
  • Auto-capture via 12 Claude Code hooks + 22 OpenCode hooks so the user "never has to call add()".
  • Coverage of Claude Code, Codex CLI, Cursor, Gemini CLI, OpenCode, OpenClaw, Hermes, pi, OpenHuman, Cline, Goose, Kilo Code, Aider, Windsurf, Roo Code, Claude Desktop via MCP plus a small set of native plugins.

Stripped of marketing, the actual gap the project fills is: a vendor-neutral, locally-hosted observation buffer for the coding-agent ecosystem, addressable from anything that speaks MCP or HTTP, with batteries-included retrieval over conversation history. That's a real gap — none of Mem0 / Letta / Zep ship as a "long-running daemon you wire to Claude Code's hooks.json" out of the box.

3 · Action — how it actually works

Reading the code, not the README. Six lenses: architecture, storage, retrieval, API surface, integrations, code quality.

3.1 Architecture at a glance

The architecture is explicit and somewhat surprising. The package itself is not a memory engine — it's a worker that registers ~50 business-logic functions against an external runtime called iii-engine, which it talks to over a local WebSocket on port 49134. From AGENTS.md:

Everything goes through registerFunction / registerTrigger / sdk.trigger() — never bypass iii-engine with standalone SQLite or in-process alternatives.

The boot sequence in src/index.ts L166-186 registers the worker and immediately wires up an in-process StateKV wrapper:

const sdk = registerWorker(config.engineUrl, {
  workerName: "agentmemory",
  invocationTimeoutMs: 180000,
  otel: { /* ... */ },
  telemetry: { project_name: "agentmemory", language: "node", framework: "iii-sdk" },
});

const kv = new StateKV(sdk);

StateKV is the thinnest possible shim — every operation is an RPC to the engine (src/state/kv.ts L6-46):

async get<T>(scope: string, key: string): Promise<T | null> {
  return this.sdk.trigger({ function_id: 'state::get', payload: { scope, key } })
}
async set<T>(scope: string, key: string, value: T): Promise<T> {
  return this.sdk.trigger({ function_id: 'state::set', payload: { scope, key, value } })
}

The implication: a single kv.set() is a WebSocket round-trip plus a SQLite write inside another process. On a write-heavy hook (e.g. Claude Code firing PostToolUse after every tool call), latencies compound and timeouts must be defended against. The repo does this with a top-level safety net (src/index.ts L112-129) that swallows unhandledRejections rather than letting one timed-out SDK call kill the daemon. Honest comment, but the pattern admits how fragile the timing is under load.

The actual call graph after registration:

The viewer runs on restPort + 2 (default :3113) as a fourth surface, served from src/viewer/server.ts.

This means agentmemory is not a self-contained npm install. Running npx @agentmemory/agentmemory triggers the iii-console installer (the workaround for which dominated the v0.9.19 changelog) and stands up two processes. The README's "External deps: None (SQLite + iii-engine)" line in the vs-competitors table is technically true only if you don't count "another networked process from the same author's company" as an external dependency. Most reviewers would.

3.2 Storage & persistence

State lives in ./data/state_store.db, a SQLite file managed by iii-engine's StateModule. agentmemory only sees logical KV scopes — namespaces like KV.memories, KV.observations(sessionId), KV.commits — defined in src/state/schema.ts.

The vector index is in-memory only, persisted lazily. From src/state/vector-index.ts L23-29:

export class VectorIndex {
  private vectors: Map<string, { embedding: Float32Array; sessionId: string }> = new Map();
  add(obsId: string, sessionId: string, embedding: Float32Array): void {
    this.vectors.set(obsId, { embedding, sessionId });
  }

Search is brute-force cosine over every stored vector (L35-63) — no ANN index, no HNSW, no IVF. For a few hundred thousand 384-dim vectors that's fine on a laptop; past a few million it will not be. This is the right choice for the stated target ("a personal coding agent on your machine") but disqualifies the daemon from team-scale archives without additional engineering.

Where the persistence layer impresses is in defense. The boot path refuses to start if the persisted vector index's dimensionality doesn't match the configured embedding provider (src/index.ts L368-403):

throw new Error(
  `[agentmemory] Refusing to start: persisted vector index has ` +
    `${mismatches.length} of ${loaded.vector.size} vectors with the ` +
    `wrong dimension. Active provider (${embeddingProvider?.name}) ` +
    `declares ${activeDim}; dimensions seen on disk: ${distinct}. ` +
    /* ... three-line recovery instructions ... */
);

That is exactly the kind of failure mode — silent cross-dim cosine returning 0, recall mysteriously degrades, nobody notices — that bites every team that swaps embedding models. Catching it at boot with an actionable error is a small thing that signals maturity. Most v0.x projects would not bother.

3.3 Embedding & retrieval

Embeddings are pluggable across eight providers in src/providers/embedding/: OpenAI, Cohere, Voyage, Gemini, OpenRouter, MiniMax-compatible, CLIP for images, and a local provider built on @xenova/transformers + ONNX (which is why the package has heavy optionalDependencies like onnxruntime-node and @xenova/transformers in package.json L67-72).

Detection order is documented in .env.example L62-68: EMBEDDING_PROVIDER override → Gemini → OpenAI → Voyage → Cohere → OpenRouter → local. Default is local all-MiniLM-L6-v2 (384-dim) — free, offline, and what the benchmark numbers were measured against.

The retrieval pipeline in src/state/hybrid-search.ts is the most thoughtful code in the repo. Three streams are merged using Reciprocal Rank Fusion with k=60 (L208-219):

const combined = Array.from(scores.entries()).map(([obsId, s]) => ({
  /* ... */
  combinedScore:
    effectiveBm25W   * (1 / (RRF_K + s.bm25Rank)) +
    effectiveVectorW * (1 / (RRF_K + s.vectorRank)) +
    effectiveGraphW  * (1 / (RRF_K + s.graphRank)),
}));

Weights are dynamically renormalised when a stream produces no results (L194-206), so a missing embedding provider doesn't drag every BM25 hit's score toward zero. After fusion there is session-level diversification (max 3 results per session) and an optional rerank stage gated by RERANK_ENABLED=true. The graph stream switched from BFS to Dijkstra over weighted edges in v0.9.19 (PR #463) after a contributor benchmarked the BFS edge-count semantics — the kind of upgrade that suggests the project actually cares about retrieval quality, not just headline numbers.

The benchmark methodology itself is more honest than the README implies. benchmark/LONGMEMEVAL.md explicitly states:

We do NOT claim these as "LongMemEval scores" — they are retrieval-only evaluations on the LongMemEval-S haystack.

…and includes a reproducibility recipe (HuggingFace dataset + two npx tsx commands). It also discloses that the closest peer, MemPalace's vector-only baseline, slightly beats agentmemory's hybrid at R@5 (96.6% vs 95.2%). That nuance does not survive the journey to the top-of-README badges, which assert 95.2% retrieval R@5 next to 92% fewer tokens without context. Worse, the vs-competitors table places agentmemory's LongMemEval R@5 (95.2%) next to Mem0's LoCoMo R@5 (68.5%) and Letta's LoCoMo R@5 (83.2%) — different benchmarks on different datasets, which is the kind of comparison a reviewer can't endorse.

3.4 Public API surface

agentmemory has not one public API but four:

  • REST — 124 endpoints under /agentmemory/*, registered in src/triggers/api.ts.
  • MCP — 53 tools (8 visible by default, all behind AGENTMEMORY_TOOLS=all), 6 resources, 3 prompts; registered in src/mcp/server.ts (1,730 lines) and src/mcp/tools-registry.ts (948 lines).
  • iii-sdk function calls — the underlying mem::* worker functions, callable from any iii-sdk client (Node or Python).
  • Hook scripts — .mjs files in plugin/scripts/ that read JSON from stdin and POST to the REST API.

The core verbs are memory_save, memory_recall, memory_smart_search, memory_compress_file, memory_file_history. The MCP handler for memory_save (src/mcp/server.ts L158-187) is a representative pattern: hand-rolled type checks (typeof args.content !== "string"), CSV-to-array parsing for concepts / files, and delegation via sdk.trigger({ function_id: "mem::remember", ... }).

A representative backend function is src/functions/remember.ts. It has more semantics than the name suggests — content-level deduplication via Jaccard similarity (L57-70):

for (const existing of existingMemories) {
  if (existing.isLatest === false) continue;
  const similarity = jaccardSimilarity(lowerContent, existing.content.toLowerCase());
  if (similarity > 0.7) {
    supersededId      = existing.id;
    supersededVersion = existing.version ?? 1;
    supersededMemory  = existing;
    break;
  }
}

…with a versioned supersedes chain, BM25 indexing inside the same call (with a comment explaining why, citing #257), a fire-and-forget vector-index update, and a cascade trigger if anything was superseded. That's a thoughtful implementation. The Jaccard threshold of 0.7 is hard-coded with no configuration, which is the kind of magic number a production user will want to tune.

Two minor API issues worth knowing about

Schema validation is hand-rolled at the MCP and REST boundaries despite zod ^4.0.0 being a direct dependency. Only two internal files import it — src/eval/schemas.ts and src/eval/validator.ts; the user-facing API does not. A Zod-first refactor would shrink the MCP server by hundreds of lines and remove a long tail of edge cases.

The Python quickstart is documentation drift. examples/python/quickstart.py L19-28 passes project and title fields to mem::remember, but the registered function signature accepts neither (title is auto-derived from content.slice(0, 80)). The call will succeed silently with the user's intended title discarded. A small but representative example of the rate of feature/doc drift at this release cadence.

3.5 Integrations (Claude Code, Codex, OpenCode, MCP, …)

The integration story is genuinely the project's commercial advantage. Native plugins ship in plugin/ for Claude Code (12 hooks), Codex (6 hooks), and OpenCode (22 hooks shipped in v0.9.21 per PR #237). Anything that speaks MCP can be wired with agentmemory connect <agent> (handlers in src/cli/connect/ — claude-code, codex, cursor, gemini-cli, hermes, openclaw, openhuman, pi).

The hook scripts under plugin/scripts/ are standalone Node.js — no iii-sdk dependency — that read JSON from stdin, POST to the REST API with a timeout, and exit. Per the AGENTS.md coding standards:

Always use try/catch with AbortSignal.timeout() for best-effort calls.

That's the right shape: best-effort, non-blocking, isolated from the daemon process. The cost is that the surface area is enormous — Claude Code's PostToolUse fires after every tool call, so a typical session generates dozens of HTTP requests against the daemon. Issue #539 ("PostToolUse hook reads data.tool_output but Claude Code sends tool_response") is a live example of how brittle that hook contract is when the upstream agent's payload shape changes.

3.6 Code-quality observations

Reading 140+ source files yields a consistent picture:

Test culture is real

95 test files, 1,067 passing tests on the v0.9.21 tag per the changelog. Vitest, vi.mock("iii-sdk"), separate unit vs integration paths. Adversarial cases are real: test/vector-index-dimensions.test.ts, test/viewer-security.test.ts, test/stop-hook-recursion-guard.test.ts, test/replay-sensitive.test.ts.

Comments violate the rules — usefully

AGENTS.md says "no code comments explaining WHAT — use clear naming instead", but the codebase is densely commented with why. The justification block before the unhandledRejection handler and the eight-line essay explaining why rebuildIndex is fired-and-forgotten are the kind of comments senior engineers love. Policy and practice are out of step; practice wins.

Concurrency hygiene is decent

Named mutexes via withKeyedLock("mem:remember", ...) (functions/remember.ts L52) protect the read-then-write deduplication path. Vector-index writes are guarded by vectorIndexAddGuarded. Validation, by contrast, is per-handler and ad-hoc — easy to add a new endpoint that gets one check wrong.

Bus factor is one

GitHub's contributor stats show rohitg00 with 320 commits out of ~362 total across 29 contributors, with the next-most-prolific (Tanmay-008) at 9. Combined with any / as unknown as T casts in load-bearing places like VectorIndex.restoreFrom, this is a project that lives or dies with one person's calendar.

4.2 Testing & CI

CI is in .github/workflows/ci.yml and runs the test suite on Node 20 and Node 22 on Ubuntu only:

- run: npm install --package-lock-only --legacy-peer-deps --no-audit --no-fund
- run: npm ci --legacy-peer-deps --no-audit --no-fund
- run: npm run build
- run: npm test

The two-step --package-lock-only then npm ci dance exists because lockfiles are deliberately gitignored at the repo level — the team generates a fresh lockfile on every CI run. This is the design choice that motivates issue #540: "no lockfile makes supply chain attacks more likely". The author's choice is defensible (you avoid lockfile churn in PRs, you always test against the latest patch versions of dependencies) but it has real downsides: every install resolves a fresh dependency tree, every contributor potentially gets a different bundle, and a supply-chain attack on any transitive dep is detected only after it has shipped to a user. For a published CLI consumed via npm install -g, this is a meaningful risk surface that the project should address.

Compounding this: there is no Windows or macOS CI matrix — and the open issues confirm cross-platform regressions (#557 "unsupported OS when installing on Windows 11", PR #487 "Windows hook scripts quote plugin paths correctly" — Windows users with spaces in their username had every hook crash). PR #556, still open at time of writing, would add a cross-platform matrix.

4.4 Gotchas & risks (ranked)

Show-stoppers for shared deployments

  • No committed lockfile. Issue #540. Every fresh global install resolves dependencies live. Combined with heavy native optional deps (onnxruntime-node), a non-trivial supply-chain risk.
  • Zero-auth by default on :3111. The auth wrapper short-circuits when AGENTMEMORY_SECRET is unset (triggers/api.ts L37-47 and mcp/server.ts L51). Viewer at :3113 is similarly default-open.
  • Tight coupling to iii-sdk version drift. Issue #555: iii-sdk@0.11.6 breaks routing under the ^0.11.2 pin. Without a lockfile, this fault can hit at any npm install.
  • PII / secret scrubbing is opt-in. The regex-based redactor in functions/privacy.ts only runs when mem::privacy is explicitly invoked. Leaked keys are persisted to memory by default.

Live bugs & drift

  • Marketing comparisons mix benchmarks. The vs-competitors table puts LongMemEval-S R@5 next to Mem0's and Letta's LoCoMo R@5. Different datasets, different tasks.
  • Live bugs in advertised features. #553 "OpenCode plugin exposes only 8 tools instead of advertised 51"; #559 "agentmemory Graph empty"; #544 "viewer shows 0 memories" — all one-day-old as of writing.
  • Documentation drift. The Python quickstart calls mem::remember with parameters the function silently discards. The "no comments" rule is widely (and helpfully) ignored.

6 · Who should use this?

Buyer-persona guidance. Two clear yeses, two clear nos.

✓ Solo developers on Claude Code / Codex / OpenCode

This is the sweet spot. npx @agentmemory/agentmemory once, agentmemory connect claude-code once, get persistent observation history across sessions. Cost is ~$0 with local embeddings.

✓ Engineers studying MCP-based memory patterns

The hook scripts, the RRF retrieval implementation, and the iii-engine-as-runtime pattern are all worth reading even if you don't adopt the package. Apache-2.0 license makes excerpting straightforward.

✗ Teams wanting shared multi-developer memory

Wait. The lack of a multi-agent scope (open request #554), zero-auth defaults, and SQLite-on-a-laptop storage mean today's release is wrong for this use case. Re-evaluate after 1.0 + an AGENT_ID scope lands.

✗ Anyone needing a memory library

Use Mem0 or Letta. agentmemory is not architecturally a library — there is no npm install agentmemory; import { Memory }. It's a daemon you talk to over REST or MCP.

7 · Recommendations for the maintainer

In rough order of impact. Three of these have open issues already waiting for them.

Commit a lockfile.

The CI workaround is clever but doesn't help the millions of npm installs that happen outside CI. Even a weekly auto-PR to update package-lock.json is better than no lockfile. Closes the supply-chain concern from #540.

Decouple from iii-sdk minor versions.

Pin the iii-sdk dependency exactly (or vendor the small surface area you use). Issue #555 is otherwise going to recur every time iii-sdk ships a patch.

Stop comparing across benchmarks in marketing.

Drop the LongMemEval vs LoCoMo cross-references from the README. Publish a head-to-head LongMemEval run for Mem0 and Letta if you want a real comparison; otherwise let benchmark/LONGMEMEVAL.md's honest framing speak.

Make zero-auth opt-in rather than opt-out.

Or at least bind to 127.0.0.1 only by default and require an explicit AGENTMEMORY_BIND=0.0.0.0. Today's defaults assume a single-user laptop and won't survive contact with a developer running this on a shared dev box.

Add a Windows + macOS CI matrix.

PR #556 is sitting open. The bug count from Windows users in the issue tracker is the receipt.

Refactor MCP / REST schema validation to Zod.

Zod is already a dependency. The 1,730-line src/mcp/server.ts is the single largest file in the repo, mostly hand-rolled typeof checks. A Zod-first refactor would shrink it dramatically, remove a class of bugs, and auto-generate the MCP inputSchema blocks currently maintained by hand.

Cut a 1.0.

The feature surface is already enormous and velocity is high — but staying at v0.9.x indefinitely communicates "API may change at any time", which is incompatible with the production positioning of the marketing materials.
